## 1. Architecture Overview & Key Decisions

### 1.1 Core Architectural Model

The arena is a **dedicated subsystem** within the existing NanoClaw process, not a separate service. It registers as a new channel type in `src/channels/registry.ts` and hooks into the existing router. This keeps deployment simple and avoids IPC complexity between processes.

Each of the 5 bots is a **separate Telegram bot** (separate token, separate bot identity) all added to the same group (`-1003935516896`). They share no conversation state with each other but all run within the same Node.js process, each managed by an `ArenaBot` instance.

**Why a single process?** The existing architecture is channel-based within one process. Splitting to 5 processes would require IPC coordination for session tracking, grading, and shared SQLite writes — adding complexity with no benefit at this scale.

### 1.2 Key Architectural Decisions

**Decision 1: Polling vs. Webhooks**

Use **long polling** for all 5 arena bots.

Reasoning:
- Existing Telegram channel skill almost certainly uses polling (typical for Node.js bots without a public HTTPS endpoint)
- Webhooks require 5 separate HTTPS endpoints with valid TLS, adding infrastructure complexity
- Polling is simpler to manage, restart, and debug
- At arena scale (experimental, single user/small group), polling latency (1–2s) is acceptable
- Each bot gets its own `TelegramPoller` instance with independent offset tracking

**Decision 2: Arena Message Identification**

Arena messages are identified by **group chat ID** (`-1003935516896`). The main router checks `chat_id` on every incoming Telegram message. If it matches the arena group, the message is handed to `ArenaOrchestrator` instead of the normal routing pipeline.

This is clean, requires no message tagging, and is robust — the arena group ID is the discriminator.

**Decision 3: Conversation History Storage**

Each bot maintains its **own conversation history in SQLite** (the `arena_conversations` table), not in memory. This survives process restarts and enables the conversation tree logging requirement. In-memory caching of recent history is fine for performance (last N messages per bot per session thread).

**Decision 4: LiteLLM Fallback Disabling**

Arena requests to LiteLLM include the header `x-litellm-disable-fallbacks: true` (or equivalent LiteLLM parameter `fallbacks: []` in the request body). This is per-request, not a global config change, so normal NanoClaw requests are unaffected.

**Decision 5: Session Model**

A **session** is the unit of comparison. When an untagged broadcast message arrives, a `session_id` (UUID) is created. All 5 parallel responses are tagged with this session_id. Subsequent replies to a bot's response are tagged with the same session_id but marked as `is_followup: true` — they're in the conversation log but excluded from automated side-by-side grading (which only compares the initial broadcast exchange).

**Decision 6: Cost Attribution**

LiteLLM logs spend to Postgres. The arena logger queries the LiteLLM Postgres spend table after each response, matching by a custom `metadata.arena_session_id` tag passed in every arena request. This gives accurate cost per response.

**Decision 7: Bot Username → Model Mapping**

Each bot has a known Telegram username (set when the bot is created). The mapping is stored in config:

```
@ArenaDeepSeek    → deepseek-v3.2
@ArenaKimi        → kimi-k2.5
@ArenaMiniMax     → minimax-m2.5
@ArenaQwenCoder   → local-coder (Ollama)
@ArenaGemma       → local-general (Ollama)
```

Mentions are parsed from message entities to determine targeted routing.

### Executive Summary

This plan outlines the end-to-end implementation of the NanoClaw Model Arena, a feature designed for rigorous, parallel comparison of five distinct AI models within a dedicated Telegram group. A daily automated job will use Claude Sonnet to grade model responses, and a weekly reporting job will synthesize these grades into actionable insights delivered via "The Dude," the main assistant.

The implementation is broken into four distinct, logical phases, starting with foundational setup and culminating in the user-facing reporting system. This approach minimizes risk and allows for iterative development and testing.

### 1. Architecture & Core Logic

The Arena will be implemented as a specialized, high-priority "skill" that exclusively handles messages from the designated Arena Telegram group. This isolates the experimental logic from the core NanoClaw system, ensuring stability and maintainability.

**Architectural Flow:**

1.  **Ingestion:** The existing Telegram channel (`src/channels/telegram.ts`) receives an incoming message.
2.  **Dispatch:** The channel checks the `chat.id` of the message. If it matches the Arena `chat_id` (`-1003935516896`), it bypasses normal skills and forwards the entire message context to a new `ArenaSkill`.
3.  **ArenaSkill Processing (`src/skills/arena.skill.ts`):**
    *   **Routing Logic:** The skill first determines the message type based on the rules:
        *   **Reply:** If `message.reply_to_message` exists, it identifies the bot that sent the original message and targets only that model.
        *   **Targeted:** If `message.entities` contain `@mentions` corresponding to the arena bots, it targets only the mentioned models.
        *   **Broadcast:** If neither of the above is true, it targets all five configured models.
    *   **Session Management:** For a new broadcast message, a new `session_id` (e.g., a UUID) is generated. This ID will be used to group the initial prompt and all parallel responses for later analysis. Subsequent replies or targeted messages within the same logical exchange will be linked via conversation history but may not share the same initial `session_id`.
    *   **Context Building:** For each targeted model, the skill queries the `arena_conversations` table to construct its unique, independent conversation history.
    *   **Parallel Invocation:** The skill uses `Promise.allSettled` to dispatch requests to all targeted models simultaneously. Each request includes the prompt, the model-specific history, and a directive to the LiteLLM proxy.
    *   **Response Handling:** As each promise resolves, the skill formats the response (e.g., `[DeepSeek V3.2]:\n\n{response_text}`) and sends it to the Telegram group. It also logs the complete transaction details to the `arena_logs` and `arena_conversations` tables.
4.  **Model Interaction:**
    *   **Cloud Models (DeepSeek, Kimi, MiniMax):** Requests are sent to the LiteLLM proxy on port 4000. LiteLLM handles authentication and forwards the request to the respective cloud provider.
    *   **Local Models (Qwen, Gemma):** Requests are sent to LiteLLM, which is configured to route `local-coder` and `local-general` to the local Ollama instance.
    *   **Tool Use:** The `ArenaSkill` will manage tool calls. For local models, any tool call execution will be wrapped in a `try...catch` block. On failure, the error, tool name, and input will be logged, and a "soft-fail" message will be returned to the model so it can continue its generation.

```sql
CREATE TABLE IF NOT EXISTS arena_sessions (
  id                INTEGER PRIMARY KEY AUTOINCREMENT,
  session_id        TEXT NOT NULL UNIQUE,
  prompt_text       TEXT NOT NULL,           -- The original user message
  user_id           INTEGER,
  username          TEXT,
  routing_type      TEXT NOT NULL,           -- 'broadcast' | 'targeted' | 'reply'
  targeted_models   TEXT,                    -- JSON array of model slugs if targeted, null for broadcast
  models_responded  TEXT,                    -- JSON array of models that actually responded
  created_at        TEXT NOT NULL DEFAULT (datetime('now')),
  graded            INTEGER NOT NULL DEFAULT 0,
  graded_at         TEXT
);

CREATE INDEX idx_arena_sessions_created ON arena_sessions(created_at);
CREATE INDEX idx_arena_sessions_graded ON arena_sessions(graded);
```

### 2.3 `arena_conversations` Table

Full conversation trees for manual review. One row per message (user or bot) in any arena thread. Turn-based schema enables incremental appends and efficient querying.

```sql
CREATE TABLE IF NOT EXISTS arena_conversations (
```

### 2.4 `arena_grades` Table

Grading output from the daily Claude Sonnet evaluation job.

```sql
CREATE TABLE IF NOT EXISTS arena_grades (
  id                    INTEGER PRIMARY KEY AUTOINCREMENT,
  session_id            TEXT NOT NULL,
  model                 TEXT NOT NULL,
  arena_log_id          INTEGER NOT NULL REFERENCES arena_logs(id),
  
  -- Rubric scores (1–10 scale for granularity)
  score_correctness     REAL,
  score_completeness    REAL,
  score_code_quality    REAL,    -- null if no code in response
  score_clarity         REAL,
  score_tool_use        REAL,    -- null if no tools used
  score_overall         REAL,    -- weighted composite
  
  -- Grader metadata
  grader_model          TEXT NOT NULL DEFAULT 'claude-sonnet-4-5',
  grader_reasoning      TEXT,    -- Claude's explanation for scores
  graded_at             TEXT NOT NULL DEFAULT (datetime('now')),
  grading_job_id        TEXT,    -- UUID for the grading run that produced this
  
  UNIQUE(arena_log_id)           -- One grade per log entry; enables idempotent re-runs
);

CREATE INDEX idx_arena_grades_session ON arena_grades(session_id);
CREATE INDEX idx_arena_grades_model ON arena_grades(model);
CREATE INDEX idx_arena_grades_graded_at ON arena_grades(graded_at);
```

### 2.5 `arena_grading_jobs` Table

Tracks daily grading job runs for crash recovery and auditability.

```sql
CREATE TABLE IF NOT EXISTS arena_grading_jobs (
  id                    INTEGER PRIMARY KEY AUTOINCREMENT,
  job_id                TEXT NOT NULL UNIQUE,
  started_at            TEXT NOT NULL,
  completed_at          TEXT,
  sessions_graded       INTEGER DEFAULT 0,
  sessions_failed       INTEGER DEFAULT 0,
  status                TEXT NOT NULL DEFAULT 'running', -- 'running' | 'completed' | 'failed'
  error                 TEXT
);
```

**Schema design rationale:** The `arena_sessions` table is a materialized summary. While its data could be derived from `arena_logs`, the grading job, report generator, and session lookup all need this metadata frequently. A cheap explicit table avoids repeated aggregation queries. Similarly, `arena_grading_jobs` enables crash recovery: on restart, the job queries ungraded sessions (`graded = 0`) and resumes, with the `UNIQUE(arena_log_id)` constraint on `arena_grades` preventing double-grading.
