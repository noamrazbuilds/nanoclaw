# NanoClaw Model Arena — Implementation Plan

## Executive Summary

The Model Arena is a parallel AI comparison system embedded in a dedicated Telegram group. Five bots respond to every untagged message simultaneously, each maintaining independent conversation history. All interactions are logged, graded daily by Claude Sonnet, and surfaced weekly (or on-demand) via The Dude. This plan covers architecture, schema, file changes, and a phased rollout.

## 1. Architecture Decisions

### 1.1 Polling vs Webhooks

**Decision: Use long-polling for all 5 arena bots.**

Reasoning:
- The existing Telegram channel skill almost certainly uses polling already (common for Node.js bots without a public HTTPS endpoint)
- Webhooks require 5 distinct public HTTPS endpoints with valid TLS certificates — significant infra overhead for an experimental feature
- Polling latency (~1–2s) is acceptable for a comparison arena; users are waiting for 5 responses anyway
- Simpler failure model: if a bot's poll loop crashes, it restarts independently without webhook re-registration
- Each bot runs its own polling loop in a separate channel instance within the existing channel registry

### 1.2 Arena Message Disambiguation

**How arena messages are distinguished from normal NanoClaw messages:**

The arena group has a fixed, known `chat_id: -1003935516896`. The router checks incoming Telegram updates' `chat.id` against this value. If it matches, the message is dispatched to the `ArenaChannel` handler, not the normal `TelegramChannel`. Normal NanoClaw messages come from different chat IDs (DMs, other groups). Zero ambiguity — it's a hard filter by chat ID.

```typescript
// src/router.ts addition
if (update.message?.chat?.id === ARENA_CHAT_ID) {
  return arenaRouter.route(update);
}
```

### 1.3 Channel Architecture

Rather than 5 completely independent processes, the arena bots run as 5 channel instances within the existing Node.js process, registered in `src/channels/registry.ts`. Each bot has:
- Its own Telegram bot token
- Its own polling loop
- Its own conversation history store (in-memory Map keyed by `chat_id + user_id`, persisted to SQLite)
- Its own LiteLLM model identifier

The `ArenaOrchestrator` (new class) coordinates broadcast sessions: it receives a message, creates a `session_id`, and fans out to all 5 `ArenaBotChannel` instances in parallel via `Promise.allSettled`.

### 1.4 Conversation History Model

Each bot maintains independent history. History is stored in SQLite (not just memory) so restarts don't lose context. The key is `(bot_id, chat_id, thread_id)` where `thread_id` is derived from the Telegram message thread if present, or defaults to the group's main thread.

**Reply-to routing:** When a user replies to a specific bot's message, the incoming update's `reply_to_message.from.id` is the bot's Telegram user ID. The arena router checks this and routes exclusively to that bot's channel. The bot appends to its own history; others are not notified.

**@mention routing:** The arena router parses `message.entities` for `mention` type, extracts the username, maps to the bot's channel, and routes only to matching bots.

### 1.5 Tool Access

All 5 bots get the same tool definitions passed in the LiteLLM request. Local models (Qwen Coder 3B, Gemma 4B) use a soft-fail wrapper: tool call responses that are malformed JSON, exceed a 10-second timeout, or return errors are caught, logged as `{ success: false, error: "...", capability_gap: true }` in `tool_calls`, and the bot continues with a degraded response rather than crashing.

### 1.6 LiteLLM Configuration

Arena requests set `fallbacks: []` and `metadata: { arena_session: true }` in the LiteLLM request body. The `metadata` field flows through to Postgres spend logs, enabling cost attribution per arena session. No fallbacks means if DeepSeek V3.2 is down, that bot's response is logged as an error — intentional for clean comparison data.

### 1.7 Grading Architecture

Daily grading runs as a task in the existing `TaskScheduler`. It:
1. Queries all broadcast sessions from the past 24h
2. Groups responses by `session_id`
3. Sends a structured prompt to Claude Sonnet (via LiteLLM, tagged `arena_grader`) with all responses side-by-side
4. Parses the structured JSON rubric scores
5. Writes to `arena_grades`

Claude Sonnet is used because it's the strongest available grader and already accessible via LiteLLM. The grader prompt is deterministic and versioned (stored in `src/arena/grader-prompt.ts`).

```sql
CREATE TABLE arena_grades (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  session_id      TEXT NOT NULL,
  bot_id          TEXT NOT NULL,
  grader_model    TEXT NOT NULL DEFAULT 'claude-sonnet-4-5',
  grader_version  TEXT NOT NULL,        -- semver of grader prompt, e.g. '1.0.0'
  score_total     REAL NOT NULL,        -- 0–100 composite
  score_correctness   REAL,             -- 0–25
  score_completeness  REAL,             -- 0–20
  score_code_quality  REAL,             -- 0–20, null if no code
  score_clarity       REAL,             -- 0–20
  score_tool_efficiency REAL,           -- 0–15
  grade_rationale TEXT,                 -- Claude's explanation JSON
  graded_at       TEXT NOT NULL DEFAULT (datetime('now')),
  FOREIGN KEY (session_id, bot_id) REFERENCES arena_logs(session_id, bot_id)
);

CREATE INDEX idx_arena_grades_session ON arena_grades(session_id);
CREATE INDEX idx_arena_grades_bot ON arena_grades(bot_id, graded_at);
```

**Score weights rationale:** Correctness (25) is weighted highest because wrong answers are categorically worse. Code quality (20) is conditionally applied — if no code is present, those points redistribute proportionally. Tool efficiency (15) is lowest because not all prompts invoke tools.

### 5.1 `ArenaOrchestrator` (src/arena/index.ts)

```typescript
class ArenaOrchestrator {
  private bots: Map<string, ArenaBotChannel> = new Map();

  async initialize() {
    for (const botConfig of ARENA_BOTS) {
      const token = await oneCLI.getSecret(botConfig.tokenSecret);
      const bot = new ArenaBotChannel(botConfig, token);
      await bot.startPolling();
      this.bots.set(botConfig.id, bot);
    }
  }

  async handleBroadcast(update: TelegramUpdate): Promise<void> {
    const sessionId = generateUUID();
    const userMessage = update.message.text;
    
    await arenaDb.insertSession({
      sessionId, userId: update.message.from.id,
      userMessage, routingType: 'broadcast',
      botCount: this.bots.size,
    });

    const results = await Promise.allSettled(
      [...this.bots.values()].map(bot => 
        bot.respondWithLogging(update, sessionId)
      )
    );

    // Log any settlement rejections as error rows in arena_logs
    results.forEach((r, i) => {
      if (r.status === 'rejected') {
        const botId = [...this.bots.keys()][i];
        arenaDb.logError(sessionId, botId, r.reason);
      }
    });
  }
}
```

### 5.2 `ArenaBotChannel` (src/arena/arena-bot-channel.ts)

Core responsibilities:
- Maintains its own `TelegramBot` instance (node-telegram-bot-api or grammy)
- Polls independently
- Sends updates to `ArenaRouter` via the existing IPC mechanism or direct call
- On `respondWithLogging()`: fetches conversation history, calls LiteLLM, sends Telegram message, logs to `arena_logs`, updates `arena_conversations`

```typescript
async respondWithLogging(update: TelegramUpdate, sessionId: string): Promise<void> {
  const history = await conversationStore.getHistory(this.config.id, update.message.chat.id);
  const startTime = Date.now();
  
  let response: LLMResponse;
  try {
    response = await this.callLiteLLM(history, update.message.text);
  } catch (err) {
    await arenaDb.insertLog({
      sessionId, botId: this.config.id, error: err.message,
      latencyMs: Date.now() - startTime, ...
    });
    return;
  }

  const sentMessage = await this.telegram.sendMessage(
    update.message.chat.id,
    response.text,
    { reply_to_message_id: update.message.message_id }
  );

  await arenaDb.insertLog({
    sessionId,
    botId: this.config.id,
    model: this.config.model,
    promptText: update.message.text,
    historyJson: JSON.stringify(history),
    responseText: response.text,
    toolCallsJson: JSON.stringify(response.toolCalls ?? []),
    tokensIn: response.usage?.prompt_tokens,
    tokensOut: response.usage?.completion_tokens,
    latencyMs: Date.now() - startTime,
    messageId: sentMessage.message_id,
    isBroadcast: 1,
  });

  await conversationStore.appendTurn(
    this.config.id, update.message.chat.id,
    update.message, sentMessage, sessionId
  );
}
```

### 5.3 `ArenaRouter` (src/arena/arena-router.ts)

Routing logic:

```typescript
class ArenaRouter {
  async handleUpdate(update: TelegramUpdate) {
    const msg = update.message;
    
    // Reaction handling
    if (update.message_reaction) {
      return this.handleReaction(update.message_reaction);
    }

    if (!msg?.text) return;

    // Determine routing type
    const mentions = this.extractMentions(msg);
    const replyTo = msg.reply_to_message;

    if (replyTo) {
      // Reply-to: route to the bot whose message was replied to
      const targetBot = this.getBotByTelegramId(replyTo.from?.id);
      if (targetBot) {
        return orchestrator.handleTargeted(update, [targetBot.id], 'reply-to');
      }
    }

    if (mentions.length > 0) {
      // Targeted: route to mentioned bots only
      const targetBots = mentions
        .map(u => this.getBotByUsername(u))
        .filter(Boolean);
      if (targetBots.length > 0) {
        return orchestrator.handleTargeted(update, targetBots.map(b => b.id), 'targeted');
      }
    }

    // Default: broadcast to all
    return orchestrator.handleBroadcast(update);
  }

  private async handleReaction(reaction: TelegramMessageReaction) {
    // Map emoji to rating: 👍 = 1, 👎 = -1
    const rating = reaction.new_reaction?.some(r => r.emoji === '👍') ? 1
                 : reaction.new_reaction?.some(r => r.emoji === '👎') ? -1 : 0;
    
    await arenaDb.updateRating(reaction.message_id, reaction.chat.id, rating);
  }
}
```

### 5.5 `ToolSoftFail` Wrapper (src/arena/tool-soft-fail.ts)

```typescript
async function executeToolWithSoftFail(
  toolName: string,
  input: unknown,
  executor: () => Promise<unknown>,
  isLocal: boolean
): Promise<ToolCallRecord> {
  const start = Date.now();
  
  if (!isLocal) {
    // Cloud models: normal execution
    const output = await executor();
    return { toolName, input, output, success: true, latencyMs: Date.now() - start };
  }

  // Local models: soft-fail mode
  try {
    const output = await Promise.race([
      executor(),
      new Promise((_, reject) => setTimeout(() => reject(new Error('TOOL_TIMEOUT')), 10000))
    ]);
    return { toolName, input, output, success: true, latencyMs: Date.now() - start, capabilityGap: false };
  } catch (err) {
    console.warn(`[Arena] Local model tool soft-fail: ${toolName}`, err.message);
    return {
      toolName, input, output: null, success: false,
      latencyMs: Date.now() - start,
      capabilityGap: true,
      error: err.message,
    };
  }
}
```

### 5.6 Cost Reconciler (src/arena/cost-reconciler.ts)

LiteLLM logs costs to Postgres asynchronously. The reconciler runs after grading:

```typescript
async function reconcileCosts(sessionIds: string[]): Promise<void> {
  // Query LiteLLM Postgres spend logs
  const costs = await pgClient.query(`
    SELECT metadata->>'bot_id' as bot_id,
           metadata->>'session_id' as session_id,
           SUM(spend) as total_cost
    FROM litellm_spendlogs
    WHERE metadata->>'arena_session' = 'true'
      AND metadata->>'session_id' = ANY($1)
    GROUP BY metadata->>'bot_id', metadata->>'session_id'
  `, [sessionIds]);

  for (const row of costs.rows) {
    await arenaDb.updateCost(row.session_id, row.bot_id, row.total_cost);
  }
}
```

**Note:** This requires LiteLLM to pass `metadata` through to spend logs. Verify this is configured in the LiteLLM proxy config (`litellm_settings.store_model_in_db: true` and appropriate metadata passthrough). If LiteLLM doesn't pass arbitrary metadata to spend logs, an alternative is to log a unique `request_id` from the LiteLLM response and join on that.

### 5.7 Daily Grading Job (src/arena/arena-grader.ts)

```typescript
class ArenaGrader {
  async runDailyGrading(): Promise<void> {
    const cutoff = new Date(Date.now() - 24 * 60 * 60 * 1000).toISOString();
    
    // Get all broadcast sessions with multiple responses from past 24h
    const sessions = await arenaDb.getBroadcastSessionsSince(cutoff);
    const sessionIds = sessions.map(s => s.session_id);

    // Reconcile costs first
    await reconcileCosts(sessionIds);

    // Group responses by session
    const grouped = await arenaDb.getLogsBySessionIds(sessionIds);

    for (const [sessionId, responses] of Object.entries(grouped)) {
      if (responses.length < 2) continue; // Skip single-response sessions (errors)
      
      const grades = await this.gradeSession(sessionId, responses);
      await arenaDb.insertGrades(grades);
    }
  }

  private async gradeSession(sessionId: string, responses: ArenaLog[]): Promise<ArenaGrade[]> {
    const prompt = buildGraderPrompt(responses); // from grader-prompt.ts
    
    const result = await fetch('http://localhost:4000/chat/completions', {
      method: 'POST',
      body: JSON.stringify({
        model: ARENA_CONFIG.graderModel,
        messages: [
          { role: 'system', content: GRADER_SYSTEM_PROMPT },
          { role: 'user', content: prompt },
        ],
        response_format: { type: 'json_object' },
        metadata: { arena_grader: true },  // Don't count grader costs in arena model costs
      }),
    });

    const parsed = JSON.parse(result.choices[0].message.content);
    return this.mapGraderOutput(sessionId, parsed);
  }
}
```

### Independent Assessment & Implementation Plan: NanoClaw Model Arena

This document outlines a comprehensive implementation plan for the NanoClaw Model Arena feature. My assessment is that this is a well-defined and highly valuable feature for systematically evaluating model performance. The plan prioritizes leveraging existing infrastructure while introducing new, isolated components to manage the arena's specific logic.

---

### 1. Executive Summary

The Model Arena will be implemented as a specialized "skill" or "channel handler" within the existing NanoClaw architecture. A new `ArenaManager` module will orchestrate all logic specific to the designated Telegram group (`-1003935516896`). This approach isolates the experimental feature, minimizing impact on the core application.

The implementation will proceed in five distinct phases:
1.  **Foundation & Setup:** Configure bots, database schemas, and environment variables.
2.  **Core Message Handling:** Implement the primary message routing logic (broadcast, targeted, reply-to).
3.  **Comprehensive Logging:** Integrate detailed logging to the new SQLite tables.
4.  **Automated Grading:** Develop the daily job to evaluate model responses using Claude Sonnet.
5.  **Reporting & Insights:** Build the weekly and on-demand reporting functionality, triggered via "The Dude".

This plan emphasizes robust logging, modular design, and automation to create a powerful, low-maintenance model evaluation pipeline.

---

### 2. Architecture & Design Decisions

#### 2.1. Telegram Bot Management: Polling vs. Webhook

*   **Decision:** Use a single webhook endpoint for all 5 bots, managed by the existing Telegram channel infrastructure.
*   **Reasoning:**
    *   **Efficiency:** A single webhook is far more resource-efficient than running five separate polling loops. It reacts to events instantly without constant checking.
    *   **Centralization:** The existing `src/channels/telegram.ts` (or equivalent) can act as a single entry point. It will receive an update, identify which bot token it corresponds to, and then pass it to the central router. This avoids duplicating bot client logic.
    *   **Scalability:** While we start with 5 bots, this architecture scales easily to N bots without adding N polling processes.

#### 2.2. Arena Message Identification & Routing

*   **Decision:** The primary mechanism for identifying an arena message will be the `chat_id`. All messages originating from `-1003935516896` will be routed to a new, dedicated `ArenaManager` skill.
*   **Reasoning:** This is the simplest and most robust method. It creates a clean separation of concerns. The main message router (`src/router.ts`) will have a simple rule: `if (message.chat.id === config.ARENA_CHAT_ID) { routeTo(ArenaManager); } else { proceedWithNormalRouting; }`. This prevents arena logic from interfering with standard NanoClaw operations.

#### 2.3. Concurrency and State Management

*   **Decision:** A single `ArenaManager` module will handle all concurrent requests to the 5 models for a given user prompt. It will use `Promise.allSettled` to dispatch requests in parallel and handle individual model failures gracefully. Conversation history will be managed in-memory with persistence to the `arena_conversations` database table.
*   **Reasoning:**
    *   `Promise.allSettled` is superior to `Promise.all` here because we want to process all model responses, even if one or more models fail. We need to log the success *and* failure cases.
    *   An in-memory cache (e.g., a `Map` object) for conversation histories, keyed by `model_name`, is sufficient for performance. This state will be built by loading from the `arena_conversations` table upon the first reply-to message in a thread and updated after each turn.

### 5. Automated Grading & Reporting Workflow

#### 5.1. Daily Grading Job (`grade_arena_sessions.ts`)
1.  **Trigger:** Cron job at ~2 AM daily.
2.  **Fetch Data:** Selects distinct `session_id`s from `arena_logs` where the timestamp is within the last 24 hours.
3.  **Process Sessions:** For each `session_id`:
    *   Fetch all associated logs (the user prompt and all model responses).
    *   Format the data into a single prompt for Claude Sonnet. This prompt will contain:
        *   The system prompt given to the models.
        *   The user's prompt.
        *   The side-by-side responses, clearly labeled by model.
        *   The detailed grading rubric.
        *   A strict instruction to return the output as a JSON object with a key for each model, containing scores and a brief critique.
4.  **Execute Grading:** Sends the prompt to Claude Sonnet.
5.  **Store Results:** Parses the JSON response from Sonnet and inserts the structured grades into the `arena_grades` table, linking back to the original `arena_logs` entry.

#### 5.2. Weekly & On-Demand Reporting (`arena_reporter.ts`)
1.  **Trigger:**
    *   **Weekly:** Cron job on Friday morning.
    *   **On-demand:** A message to "The Dude".
2.  **Data Aggregation:** The reporter function takes a date range as input. It executes SQL queries against `arena_grades` and `arena_logs` to calculate:
    *   **Overall Rankings:** Average score per model across all rubric categories.
    *   **Win Rates:** For each session, determine the model with the highest overall score. Tally these "wins" to calculate a win rate.
    *   **Cost Analysis:** `SUM(cost)` per model and a `cost_per_quality_point` metric (`SUM(cost) / SUM(overall_score)`).
    *   **Performance by Category:** Average score for each model in specific categories (e.g., who is best at `code_quality`).
    *   **Standout Examples:** Pull the top-rated response (`log_id` with the highest score) and the most contentious session (highest standard deviation in scores) from the period.
3.  **Format Output:** The aggregated data is compiled into a clean, readable Markdown report.
4.  **Delivery:** The report string is returned to the caller, which is either the scheduled task handler (for weekly reports) or "The Dude" (for on-demand), to be sent to the user.

# NanoClaw Model Arena — Implementation Plan

## Executive Summary

The Model Arena is a parallel AI comparison system embedded in a dedicated Telegram group (`-1003935516896`). Five bots respond to every untagged message simultaneously, each maintaining independent conversation history. All interactions are logged to SQLite, graded daily by Claude Sonnet, and surfaced weekly (or on-demand) via The Dude. The system is implemented as a new `src/arena/` module within the existing Node.js/TypeScript channel-based architecture, isolated behind a feature flag.

---

## 1. Architecture Decisions

### 1.1 Polling vs Webhooks

**Decision: Use long-polling, with one thin polling loop per bot token, coordinated through a single `ArenaOrchestrator`.**

Each Telegram bot has its own token and therefore requires its own connection to the Telegram API — you cannot receive updates for Bot A using Bot B's token. Five polling loops are therefore technically necessary. However, these pollers are thin: they receive updates and immediately hand them to the shared `ArenaOrchestrator` for all processing.

Reasoning for polling:
- Webhooks require 5 distinct public HTTPS endpoints with valid TLS certificates — significant infrastructure overhead for an experimental feature
- The existing Telegram channel skill almost certainly uses polling (standard for Node.js bots without a public HTTPS endpoint); the arena should match existing infrastructure patterns
- Polling latency (~1–2s) is imperceptible when users are waiting for 5 model responses
- Simpler failure model and easier to debug during experimentation
- No reverse proxy, SSL termination, or public endpoint configuration required

**Critical configuration:** All polling loops must include `allowed_updates: ["message", "message_reaction"]` to receive reaction events for the thumbs up/down rating system.

**Migration path:** If the existing infrastructure already uses webhooks, or if the system moves to webhooks later, the arena bots should use path-differentiated webhook URLs (e.g., `/arena/webhook/:botId`). The `ArenaOrchestrator` routing logic is identical either way — only the ingestion layer changes.

### 1.2 Arena Message Disambiguation

The arena group has a fixed, known `chat_id: -1003935516896`. The router checks incoming Telegram updates' `chat.id` against this value. If it matches, the message is dispatched to the `ArenaOrchestrator`, not the normal `TelegramChannel`. Normal NanoClaw messages come from different chat IDs (DMs, other groups). Zero ambiguity — it's a hard filter by chat ID.

```typescript
// src/router.ts addition
if (update.message?.chat?.id === ARENA_CONFIG.chatId) {
  return arenaOrchestrator.handleUpdate(update);
}
```

### 1.3 Channel Architecture

The arena uses a centralized orchestrator with thin per-bot pollers:

- **One `ArenaOrchestrator`** — the central coordinator, registered in `channels/registry.ts`
- **Five `ArenaBotPoller` instances** — thin wrappers that own a bot token, run a polling loop (tracking `offset` to avoid reprocessing), and forward all updates to the `ArenaOrchestrator`
- **Five `ArenaBotSender` instances** — stateless, own a bot token, used only for sending messages

```
[Telegram API] ←→ [ArenaBotPoller × 5] → [ArenaOrchestrator] → [ArenaBotSender × 5] → [Telegram API]
                                                    ↓
                                              [ArenaDB / SQLite]
```

**Fault-tolerant initialization:** If a bot fails to initialize (bad token, network error), the arena starts with the remaining bots and logs the initialization failure. The arena does not require all 5 bots to be operational.

### 1.4 Conversation History Model

Each bot maintains independent history. History is stored in SQLite via the `parent_log_id` chain in `arena_logs`, enabling stateless reconstruction. An in-memory LRU cache (1000 conversations per bot) provides fast access for active conversations, with cache misses reconstructed from the database by traversing the `parent_log_id` chain.

**Reply-to routing:** When a user replies to a specific bot's message, the incoming update's `reply_to_message.from.id` is the bot's Telegram user ID. The arena router checks this against the known bot user IDs and routes exclusively to that bot. The bot appends to its own history; others are not notified. The new `arena_logs` entry gets `parent_log_id` set to the log entry whose `telegram_message_id` matches `reply_to_message.message_id`.

**@mention routing:** The arena router parses `message.entities` for `mention` type, extracts the username, maps to the bot config (usernames stored in `ARENA_BOTS` config and verified via `getMe` API on startup), and routes only to matching bots.

**Message ordering:** When 5 bots respond in parallel, their messages arrive in non-deterministic order. Each bot's response is prefixed with its display name (e.g., `**DeepSeek V3.2:**`) so users can identify responses regardless of order. A small stagger (200ms between sends) helps with Telegram rate limits.

**User experience on failure:** If a model call fails, the bot sends a brief error message: `⚠️ [ModelName] is unavailable for this session.` This is logged as an error row in `arena_logs` with the `error` field populated.

### 1.5 Tool Access

All 5 bots get the same tool definitions passed in the LiteLLM request. Local models (Qwen Coder 3B, Gemma 4B) use a soft-fail wrapper:

```typescript
async function executeToolWithSoftFail(
  toolName: string, input: unknown, executor: () => Promise<unknown>, isLocal: boolean
): Promise<ToolCallRecord> {
  const start = Date.now();
  if (!isLocal) {
    const output = await executor();
    return { toolName, input, output, success: true, latencyMs: Date.now() - start };
  }
  try {
    const output = await Promise.race([
      executor(),
      new Promise((_, reject) => setTimeout(() => reject(new Error('TOOL_TIMEOUT')), 10_000))
    ]);
    return { toolName, input, output, success: true, latencyMs: Date.now() - start, capabilityGap: false };
  } catch (err) {
    return { toolName, input, output: null, success: false, latencyMs: Date.now() - start, capabilityGap: true, error: err.message };
  }
}
```

**Note on local models:** Ollama runs as a separate HTTP server process — calls to it are standard async HTTP requests and do not block the Node.js event loop. The timeout for local model tool calls is 10 seconds; the timeout for the entire inference request is 60 seconds.

### 1.6 LiteLLM Configuration

Arena requests set `fallbacks: []` in the request body (this is the correct mechanism — not a metadata flag) and `metadata: { arena_session: true, bot_id: "...", session_id: "..." }`. The `metadata` field flows through to Postgres spend logs for cost attribution. No fallbacks means if a cloud model is down, that bot's response is logged as an error.

**Cost reconciliation:** Depends on LiteLLM passing `metadata` to spend logs. Verify `litellm_settings.store_model_in_db: true` is configured. If metadata passthrough is unavailable, fall back to joining on the LiteLLM `request_id` returned in response headers — store this in `arena_logs.litellm_request_id`.

All bots receive an identical system prompt for pure model comparison. The prompt is stored in a versioned file (`src/arena/grader-prompt.ts` exports both the arena system prompt and the grader prompt). The version string is stored in config and logged with each session, ensuring that if the prompt changes mid-experiment, historical grades remain attributable to the prompt version they were generated under.

```typescript
class ArenaOrchestrator {
  private bots: Map<string, { poller: ArenaBotPoller; sender: ArenaBotSender }> = new Map();

  async initialize() {
    for (const botConfig of ARENA_BOTS) {
      try {
        const token = await oneCLI.getSecret(botConfig.tokenSecret);
        const poller = new ArenaBotPoller(botConfig, token);
        const sender = new ArenaBotSender(botConfig, token);
        // Verify bot identity and resolve username
        const me = await sender.getMe();
        if (me.username !== botConfig.username) {
          console.warn(`Bot ${botConfig.id} username mismatch: expected ${botConfig.username}, got ${me.username}`);
        }
        await poller.startPolling({ allowed_updates: ['message', 'message_reaction'] });
        this.bots.set(botConfig.id, { poller, sender });
      } catch (err) {
        console.error(`Failed to initialize arena bot ${botConfig.id}:`, err);
        // Continue with remaining bots
      }
    }
    console.log(`Arena initialized with ${this.bots.size}/${ARENA_BOTS.length} bots`);
  }

  async handleBroadcast(update: TelegramUpdate): Promise<void> {
    const sessionId = generateUUID();
    await arenaDb.insertSession({
      sessionId, userId: update.message.from.id,
      userMessage: update.message.text, routingType: 'broadcast',
      botCount: this.bots.size, systemPromptVersion: ARENA_CONFIG.graderVersion,
    });

    const results = await Promise.allSettled(
      [...this.bots.entries()].map(([botId, { sender }], index) =>
        this.respondWithLogging(botId, sender, update, sessionId, index)
      )
    );

    results.forEach((r, i) => {
      if (r.status === 'rejected') {
        const botId = [...this.bots.keys()][i];
        arenaDb.logError(sessionId, botId, r.reason?.message ?? 'Unknown error');
      }
    });
  }

  private async respondWithLogging(
    botId: string, sender: ArenaBotSender, update: TelegramUpdate,
    sessionId: string, sendIndex: number
  ): Promise<void> {
    const botConfig = ARENA_BOTS.find(b => b.id === botId)!;
    const history = await this.getConversationHistory(botId, update);
    const startTime = Date.now();

    let response: LLMResponse;
    try {
      response = await this.callModel(botConfig, history, update.message.text);
    } catch (err) {
      // Send error message to user
      await sender.sendMessage(update.message.chat.id,
        `⚠️ ${botConfig.displayName} is unavailable for this session.`,
        { reply_to_message_id: update.message.message_id });
      throw err; // Re-throw for Promise.allSettled to catch
    }

    // Stagger sends to avoid Telegram rate limits
    await new Promise(resolve => setTimeout(resolve, sendIndex * ARENA_CONFIG.sendStaggerMs));

    const sentMessage = await sender.sendMessage(
      update.message.chat.id,
      `**${botConfig.displayName}:**\n${response.text}`,
      { reply_to_message_id: update.message.message_id }
    );

    await arenaDb.insertLog({
      sessionId, botId, model: botConfig.model,
      chatId: update.message.chat.id, userId: update.message.from.id,
      telegramMessageId: sentMessage.message_id,
      promptText: update.message.text,
      historyJson: JSON.stringify(history),
      responseText: response.text,
      toolCallsJson: JSON.stringify(response.toolCalls ?? []),
      tokensIn: response.usage?.prompt_tokens,
      tokensOut: response.usage?.completion_tokens,
      litellmRequestId: response.requestId,
      latencyMs: Date.now() - startTime,
      isBroadcast: 1,
    });
  }
}
```

```typescript
  handleUpdate(update: TelegramUpdate) {
    // Reaction handling
    if (update.message_reaction) {
      return this.handleReaction(update.message_reaction);
    }

    const msg = update.message;
    if (!msg?.text) return;

    const replyTo = msg.reply_to_message;
    if (replyTo) {
      const targetBot = this.getBotByTelegramUserId(replyTo.from?.id);
      if (targetBot) {

    if (mentions.length > 0) {

      if (targetBots.length > 0) {
```

### 5.5 Daily Grading Job (src/arena/arena-grader.ts)

```typescript
class ArenaGrader {
  async runDailyGrading(): Promise<void> {
    const cutoff = new Date(Date.now() - 24 * 60 * 60 * 1000).toISOString();

    // Get broadcast sessions not yet graded (idempotency)
    const sessions = await arenaDb.getUngradedBroadcastSessionsSince(cutoff);

    // Reconcile costs first
    await reconcileCosts(sessions.map(s => s.session_id));

    for (const session of sessions) {
      const responses = await arenaDb.getLogsBySessionId(session.session_id);
      if (responses.length < 2) continue;

      try {
        const grades = await this.gradeSessionWithRetry(session.session_id, responses);
        await arenaDb.insertGrades(grades); // UNIQUE constraint prevents duplicates
        await arenaDb.updateSessionStatus(session.session_id, 'graded');
      } catch (err) {
        console.error(`Grading failed for session ${session.session_id}:`, err);
        await arenaDb.updateSessionStatus(session.session_id, 'failed');
      }
    }

    // Update aggregates after grading
    await this.updateAggregates(cutoff);
  }

  private async gradeSessionWithRetry(sessionId: string, responses: ArenaLog[]): Promise<ArenaGrade[]> {
    for (let attempt = 0; attempt < ARENA_CONFIG.graderMaxRetries; attempt++) {
      try {
        const prompt = buildGraderPrompt(responses);
        const result = await fetch('http://localhost:4000/chat/completions', {
          method: 'POST',
          body: JSON.stringify({
            model: ARENA_CONFIG.graderModel,
            messages: [
              { role: 'system', content: GRADER_SYSTEM_PROMPT },
              { role: 'user', content: prompt },
            ],
            response_format: { type: 'json_object' },
            metadata: { arena_grader: true },
          }),
        });
        const parsed = JSON.parse(result.choices[0].message.content);
        // Validate structure before returning
        return this.validateAndMapGraderOutput(sessionId, parsed);
      } catch (err) {
        if (attempt < ARENA_CONFIG.graderMaxRetries - 1) {
          await new Promise(r => setTimeout(r, Math.pow(2, attempt + 1) * 1000));
        } else {
          throw err;
        }
      }
    }
    throw new Error('Unreachable');
  }
}
```

**Grader prompt (src/arena/grader-prompt.ts):**

```typescript
export const GRADER_SYSTEM_PROMPT = `
You are an expert AI evaluator. You will receive a user prompt and responses from multiple AI models.
Score each response on the following rubric (return valid JSON):

- correctness (0-25): Is the response factually accurate and logically sound?
- completeness (0-20): Does it fully address all parts of the question?
- code_quality (0-20): If code is present: correctness, style, efficiency. If no code: null.
- clarity (0-20): Is the response clear, well-structured, appropriately concise?
- tool_efficiency (0-15): If tools were used: were the right tools chosen, used correctly? If no tools: null.

Return format:
{
  "grades": [
    {
      "bot_id": "...",
      "scores": { "correctness": N, "completeness": N, "code_quality": N|null, "clarity": N, "tool_efficiency": N|null },
      "total": 0-100,
      "rationale": "Brief explanation"
    }
  ]
}

Be objective. Grade the response quality, not the model brand.
If a response is an error or empty, score it 0 across all dimensions.
`;

export function buildGraderPrompt(responses: ArenaLog[]): string {
  const prompt = responses[0].prompt_text;
  const responsesText = responses.map(r =>
    `=== ${r.bot_id} ===\n${r.response_text ?? '[ERROR: No response]'}\n` +
    (r.tool_calls_json ? `[Tools used: ${r.tool_calls_json}]\n` : '')
  ).join('\n');
  return `User Prompt:\n${prompt}\n\nModel Responses:\n${responsesText}`;
}
```

| Risk | Mitigation |
|------|-----------|
| Local model instability | Soft-fail wrappers, 60s timeout, error UX messages |

### Resolved Disagreements

#### 1. Polling vs Webhooks

**Dispute:** Voice B advocated for webhooks (single endpoint, more efficient). Voices A and C advocated for polling.

**Ruling: Polling is the correct choice for this context.** Voice A's rebuttal was decisive: webhooks require 5 distinct public HTTPS endpoints with valid TLS certificates — significant infrastructure overhead for an experimental feature. The existing infrastructure almost certainly uses polling. Polling latency is irrelevant when users wait for 5 model responses. However, Voice B's point about efficiency is valid in principle — the plan includes a migration path to webhooks if the infrastructure evolves.

#### 2. Database Schema: 3 tables vs 4 tables

**Dispute:** Voice B proposed 3 tables. Voice A proposed 4 (adding `arena_sessions`). Voice C proposed 4 (adding `arena_aggregates`).

**Ruling: 4 tables is correct, but the right 4.** The final schema uses `arena_sessions`, `arena_logs`, `arena_grades`, and `arena_aggregates`. Voice A was right that `arena_sessions` provides clean metadata separation. Voice C was right that `arena_aggregates` is essential for report performance. The original `arena_conversations` table (proposed by all three in various forms) was correctly eliminated during critique — the `parent_log_id` approach in `arena_logs` is superior for conversation tree reconstruction.

#### 3. Conversation History Storage

**Dispute:** All three originally proposed some form of JSON blob storage for conversations. During critique, the `parent_log_id` self-referencing approach was identified as superior.

**Ruling: `parent_log_id` in `arena_logs` is the correct approach.** It enables SQL-level tree traversal, is stateless from the application's perspective, and eliminates the need for a separate mutable table with read-modify-write operations on JSON blobs. This was the single most valuable insight that emerged from the cross-critique process.

#### 4. Foreign Key in `arena_grades`

**Dispute:** Voice A originally used `FOREIGN KEY (session_id, bot_id) REFERENCES arena_logs(session_id, bot_id)` — which is invalid because `(session_id, bot_id)` is not unique in `arena_logs`.

**Ruling: Reference `arena_logs(id)` directly.** Voice B correctly identified this as a critical schema error. The revised schema uses `arena_log_id INTEGER NOT NULL REFERENCES arena_logs(id)` with a `UNIQUE(arena_log_id, grader_model)` constraint for idempotency.

#### 5. Grading Scale: 1-5 vs 0-100

**Dispute:** Voice C used 1-5 per dimension. Voice A used 0-25/0-20/0-15 weighted dimensions summing to 0-100.

**Ruling: The weighted 0-100 approach is superior.** It provides finer granularity, makes the weighting explicit (correctness matters more than tool efficiency), and produces a single composite score that's immediately useful for rankings and win rate calculations. The 1-5 scale loses information and requires post-hoc weighting.

#### 6. File Structure Granularity

**Dispute:** Voice B preferred fewer, more consolidated files. Voice C preferred more modular separation.

**Ruling: Moderate modularity.** The final plan uses 12 files in `src/arena/`, which is appropriate for the feature's complexity. Each file has a clear, single responsibility. This matches the existing channel-based architecture pattern without over-engineering.

### Unresolved Tensions

#### 1. Polling Architecture: Thin Pollers vs Single Dispatcher

Voice A's revised plan uses 5 thin `ArenaBotPoller` instances (one per token). Voice C argued for a "single polling loop for the arena group." The tension is that Telegram requires one connection per bot token — you genuinely need 5 connections. The question is whether these should be 5 independent polling loops or somehow multiplexed. In practice, 5 lightweight polling loops are the only viable option given Telegram's API design, but reasonable engineers might debate whether to abstract this behind a single "ArenaPollingManager" class or keep them as separate instances. **The practical difference is minimal.**

#### 2. Local Model Execution: Direct HTTP vs Container Runner

Voice B suggested using the existing `container-runner.ts` for local Ollama models to prevent blocking. Voice A rebutted that Ollama is a separate HTTP server and calls are non-blocking. Both are correct in different senses: the HTTP call is non-blocking, but if Ollama is running on the same machine, heavy inference could starve CPU from the Node.js process. **For an experimental feature with low volume, direct HTTP calls to Ollama are sufficient. If performance issues emerge, container isolation is the correct escalation path.**

#### 3. Grading Cost Tracking

None of the voices fully addressed whether the cost of running Claude Sonnet for grading should be tracked and reported separately from arena model costs. The final plan includes `metadata: { arena_grader: true }` to distinguish grading costs, but the reporting module doesn't explicitly surface grading overhead. **This is a minor gap that should be addressed during Phase 5.**

#### 4. Webhook Migration Path

The plan recommends polling but acknowledges webhooks may be preferable if the infrastructure supports them. The exact trigger for this migration (performance issues? scaling beyond one group?) is left undefined. **This is acceptable for an experimental feature — the decision can be revisited based on operational experience.**

### Debate Value Assessment

The three-round debate process produced a significantly better plan than any single response would have:

1. **Schema improvements:** The `parent_log_id` approach for conversation trees emerged entirely from cross-critique — no voice proposed it initially. The foreign key error in `arena_grades` was caught and corrected. The `arena_aggregates` table was validated as necessary through consensus.

2. **Architectural corrections:** The LiteLLM fallback mechanism was corrected from `metadata: { fallback: false }` (wrong) to `fallbacks: []` (correct) — two of three voices got this wrong initially. SQLite WAL mode was identified as necessary by the critique process.

3. **Operational completeness:** Error UX (what users see when a model fails), grading idempotency, `allowed_updates` for reactions, bot username verification via `getMe`, and message send staggering for rate limits were all gaps identified through cross-critique.

4. **Polling vs webhook decision was stress-tested:** While the debate didn't change the final recommendation, the arguments for and against were thoroughly examined, resulting in a more defensible decision with an explicit migration path.

5. **The weakest arguments were eliminated:** Over-engineered channel architecture (5 fat channel instances), JSON blob conversation storage, and incorrect LiteLLM configuration were all identified and removed through the adversarial process.
