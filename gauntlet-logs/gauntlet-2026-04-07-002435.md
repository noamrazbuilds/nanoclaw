# NanoClaw "Link-to-Audio" Skill — Implementation Plan

## Executive Summary

This plan details a new container skill (`link-to-audio`) that accepts a URL from a user via WhatsApp/Telegram, extracts the main article content, converts it to speech via chunked TTS, and returns a voice note. The design prioritizes lightweight content extraction (trafilatura over Playwright), smart text chunking at sentence boundaries, ffmpeg-based audio concatenation, and robust error handling with user feedback at each stage.

---

## 1. Content Extraction Strategy

### Recommendation: **Tiered approach — trafilatura primary, agent-browser fallback**

| Option | Pros | Cons |
|--------|------|------|
| **(a) agent-browser + Readability.js** | Handles JS-rendered SPAs, can bypass some paywalls via cookie/session | Heavy (~200MB+ memory), slow startup (2-5s), overkill for 90% of articles |
| **(b) trafilatura (Python)** | Purpose-built for article extraction, handles metadata, fast (<1s), excellent heuristics, supports fallback to readability-lxml internally | Can't handle JS-rendered content |
| **(c) HTTP + BeautifulSoup** | Minimal dependencies | Poor extraction quality, you'd be reimplementing readability |

**Decision: Use trafilatura as the primary extractor with agent-browser as an optional fallback.**

### Rationale

- trafilatura is specifically designed for main content extraction from web pages. It internally uses multiple strategies (including readability-lxml as a fallback) and handles boilerplate removal, ad stripping, and metadata extraction.
- It's pure Python, lightweight (~5MB installed), and processes most pages in under a second.
- For the ~5-10% of pages that are JS-rendered SPAs (e.g., some Medium pages behind interstitials, Bloomberg), we fall back to agent-browser: load the page in Playwright, wait for content, extract `document.body.innerHTML`, then pass *that* HTML to trafilatura.
- This avoids spinning up Playwright for the common case while still handling edge cases.

### Extraction Pipeline

```
URL → HTTP fetch (requests, with proper User-Agent + timeout)
  → trafilatura.extract(html, include_comments=False, include_tables=False)
  → if result is None or len(result) < 200:
      → agent-browser fallback: Playwright fetch → rendered HTML → trafilatura.extract()
      → if still fails: return error to user
  → Clean text (normalize whitespace, strip artifacts)
  → Return (title, text, word_count, estimated_audio_duration)
```

### Handling Paywalled Content (Best Effort)

- Set `User-Agent` to a Googlebot-like string (many paywalls serve full content to crawlers).
- Accept cookies from `12ft.io`-style bypass if available.
- If content is clearly truncated (< 200 chars with a "subscribe" CTA detected), inform the user: *"This article appears to be paywalled. I extracted what I could (~X words)."*
- Do **not** attempt aggressive paywall circumvention — this is "best effort" per the constraint.

## 2. Text Chunking and Audio Concatenation

### The Problem

OpenAI TTS has a **4096 character limit** per request. A typical long-form article is 5,000–15,000 characters. Some are 30,000+.

### Chunking Strategy

**Chunk at sentence boundaries, targeting ~3800 characters per chunk** (leaving buffer for edge cases).

```python
def chunk_text(text: str, max_chars: int = 3800) -> list[str]:
    """Split text into chunks at sentence boundaries."""
    import re
    sentences = re.split(r'(?<=[.!?])\s+', text)
    
    chunks = []
    current_chunk = ""
    
    for sentence in sentences:
        # If a single sentence exceeds max_chars, split at clause/word boundaries
        if len(sentence) > max_chars:
            if current_chunk:
                chunks.append(current_chunk.strip())
                current_chunk = ""
            # Sub-split long sentences at commas, semicolons, or word boundaries
            chunks.extend(_split_long_sentence(sentence, max_chars))
            continue
        
        if len(current_chunk) + len(sentence) + 1 > max_chars:
            chunks.append(current_chunk.strip())
            current_chunk = sentence
        else:
            current_chunk = f"{current_chunk} {sentence}" if current_chunk else sentence
    
    if current_chunk.strip():
        chunks.append(current_chunk.strip())
    
    return chunks
```

**Why 3800 not 4096?** Safety margin. Some TTS APIs count characters differently (e.g., multi-byte Unicode). 3800 gives us headroom without wasting capacity.

### Audio Generation and Concatenation

```
For each chunk:
  → Call tts.py with chunk text → /tmp/link2audio_{job_id}_chunk_{i}.mp3
  → Verify output file exists and is > 0 bytes

After all chunks:
  → Create ffmpeg concat file: /tmp/link2audio_{job_id}_filelist.txt
  → ffmpeg -f concat -safe 0 -i filelist.txt -c copy output.mp3
  → If total size > platform limit: compress with ffmpeg (lower bitrate)
  → Clean up chunk files
```

**Why `-c copy` (stream copy)?** All chunks come from the same TTS engine with identical encoding parameters. Stream copy is instant and lossless — no re-encoding needed.

### Parallel TTS Generation

For articles with many chunks (>5), generate TTS in parallel (up to 3-4 concurrent requests) to reduce wall-clock time. Use `asyncio.gather` or `concurrent.futures.ThreadPoolExecutor`.

```python
MAX_CONCURRENT_TTS = 4  # Respect rate limits

async def generate_all_chunks(chunks, voice, backend, job_id):

    async def generate_one(i, chunk):
        async with semaphore:
            return await generate_tts_chunk(chunk, voice, backend, job_id, i)
    
    tasks = [generate_one(i, c) for i, c in enumerate(chunks)]
    return await asyncio.gather(*tasks)
```

## 3. Trigger Format

### Recommendation: **Structured command with natural language fallback**

**Primary triggers:**
- `!listen <URL>` — canonical command
- `!read <URL>` — alias
- `!listen <URL> voice:shimmer` — with voice override
- `!listen <URL> backend:elevenlabs` — with backend override

**Natural language detection (secondary):**
- Messages containing a URL + intent keywords: *"read this article"*, *"convert to audio"*, *"listen to this"*, *"make this a voice note"*
- Pattern: `(read|listen|audio|voice|speak|narrate)\b.*https?://` or `https?://\S+.*(read|listen|audio|voice|speak|narrate)`

### Implementation

```python
import re

TRIGGER_PATTERNS = [
    # Explicit commands
    r'^[!/](listen|read|audio)\s+(https?://\S+)',
    # Natural language with URL
    r'(?:read|listen to|convert|narrate|make audio)\s+(?:this\s+)?(?:article\s+)?(?:from\s+)?(https?://\S+)',
    r'(https?://\S+)\s+.*(?:read|listen|audio|voice note|narrate)',
]

def parse_trigger(message: str) -> dict | None:
    """Returns {url, voice, backend} or None if not a link-to-audio request."""
    for pattern in TRIGGER_PATTERNS:
        match = re.search(pattern, message, re.IGNORECASE)
        if match:
            url = match.group(match.lastindex)  # URL is always last capture group
            voice = extract_voice_override(message)
            backend = extract_backend_override(message)
            return {"url": url, "voice": voice, "backend": backend}
    return None
```

### Why this approach?

- Explicit commands (`!listen`) are unambiguous and fast to parse — power users will prefer these.
- Natural language fallback covers casual users who just paste a link and say "read this."
- We avoid false positives by requiring both a URL and an intent keyword for natural language triggers.
- The explicit command form makes it easy to document and discover.

### Implementation

```python
VOICE_PATTERN = r'voice[:\s]+(\w+)'
BACKEND_PATTERN = r'(?:backend[:\s]+|use\s+)(\w+)'

KNOWN_BACKENDS = {"openai", "elevenlabs", "piper", "local", "robotic"}
OPENAI_VOICES = {"alloy", "echo", "fable", "onyx", "nova", "shimmer"}

def extract_voice_override(message: str) -> str | None:
    match = re.search(VOICE_PATTERN, message, re.IGNORECASE)
    if match:
        return match.group(1).lower()
    # Also detect "in X's voice" pattern
    match = re.search(r"in\s+(\w+)(?:'s)?\s+voice", message, re.IGNORECASE)
    if match and match.group(1).lower() in OPENAI_VOICES:
        return match.group(1).lower()
    return None

def extract_backend_override(message: str) -> str | None:
    match = re.search(BACKEND_PATTERN, message, re.IGNORECASE)
    if match and match.group(1).lower() in KNOWN_BACKENDS:
        return match.group(1).lower()
    return None
```

### **1. Content Extraction Approach**

The choice of content extraction method is critical, balancing the need to handle modern, JavaScript-driven websites against the performance cost of a full browser engine.

**Recommendation:** A hybrid, two-stage approach.

*   **Stage 1 (Primary): Python `trafilatura` library.**
    *   **Description:** Use the `trafilatura` Python library to perform a lightweight HTTP GET request for the URL and extract the main article text.     *   **Reasoning:**
        *   **Performance:** This method is extremely fast and has a low memory footprint. It avoids spinning up a full browser instance, which can take several seconds and consume significant resources. For the majority of standard articles and blogs, this will be sufficient and provide a near-instantaneous response for the extraction step.
        *   **Effectiveness:** `trafilatura` is a state-of-the-art library that often outperforms older solutions like `readability-lxml`. It's robust and well-maintained.
        *   **Simplicity:** It's a pure Python dependency, easy to add to the container's `requirements.txt`.

*   **Stage 2 (Fallback): `agent-browser` (Playwright).**
    *   **Description:** If Stage 1 fails to return meaningful content (e.g., the extracted text length is below a certain threshold, or the HTTP response body suggests it's a JS-loader shell), the skill will then invoke `agent-browser`. It will navigate to the page, wait for it to render, and then execute a script to extract the main content, potentially using Mozilla's Readability.js library injected into the page context.
    *   **Reasoning:**
        *   **Robustness:** This handles the complex cases: single-page applications (SPAs), sites that load content dynamically via JavaScript, and pages behind simple "I am not a robot" checks that Playwright can often handle.
        *   **Justified Cost:** The high cost of launching a browser is justified here because the simpler, faster method has already failed. It acts as a powerful fallback rather than the default, ensuring the best possible chance of success without penalizing performance on simple tasks.

**Implementation Logic:**

1.  Attempt extraction with `trafilatura`.
2.  Check the result. If extracted text is > 250 characters, proceed.
3.  If text is < 250 characters, trigger the `agent-browser` fallback.
4.  If `agent-browser` also fails, report an extraction failure to the user.

This hybrid model provides the speed of a lightweight library for the 80% case and the power of a full browser for the 20% case, offering the best overall reliability and user experience.

### **2. Handling the 4096-Character TTS Limit**

This requires a systematic approach to chunking text, generating audio for each chunk, and concatenating the results into a single file.

**Recommended Process:**

1.  **Text Pre-processing:** After extracting the full article text, clean and normalize it. Replace multiple newlines with a single paragraph break marker (e.g., `\n\n`) and normalize whitespace.

2.  **Smart Chunking:**
    *   First, split the entire text into paragraphs based on the `\n\n` delimiter.
    *   Iterate through the paragraphs, assembling them into a "chunk".
    *   Before adding a new paragraph to the current chunk, check if the combined length would exceed the 4096 character limit.
    *   If it exceeds the limit, the current chunk is complete. Start a new chunk with the new paragraph.
    *   **Edge Case:** If a single paragraph itself is longer than 4096 characters, split that paragraph at the last sentence-ending punctuation (`.`, `?`, `!`) before the limit.

3.  **Audio Generation:**
    *   For each text chunk, call the `tts.py` script to generate an audio file (e.g., `temp_audio_0.mp3`, `temp_audio_1.mp3`, etc.) in a temporary directory. This should be done sequentially to avoid overwhelming the TTS API with parallel requests.

4.  **Concatenation with `ffmpeg`:**
    *   Once all audio chunks are generated, create a text file (`filelist.txt`) containing the path to each temporary audio file, one per line, in order.
        ```
        file 'temp_audio_0.mp3'
        file 'temp_audio_1.mp3'
        file 'temp_audio_2.mp3'
        ```
    *   Execute an `ffmpeg` command to concatenate these files without re-encoding, which is extremely fast and preserves quality.
        ```bash
        ffmpeg -f concat -safe 0 -i filelist.txt -c copy final_output.mp3
        ```
    *   The `-c copy` flag is crucial for performance as it stream-copies the audio data directly.

5.  **Cleanup:** After successful concatenation, delete the temporary audio chunks and the `filelist.txt`.

## 2. Text Chunking and Audio Concatenation

### The Problem

OpenAI TTS has a **4096 character limit** per request. A typical long-form article is 5,000–15,000 characters. Some are 30,000+.

### Chunking Strategy

**Chunk at sentence boundaries, targeting ~3800 characters per chunk** (leaving buffer for edge cases).

```python
def chunk_text(text: str, max_chars: int = 3800) -> list[str]:
    """Split text into chunks at sentence boundaries."""
    import re
    sentences = re.split(r'(?<=[.!?])\s+', text)
    
    chunks = []
    current_chunk = ""
    
    for sentence in sentences:
        # If a single sentence exceeds max_chars, split at clause/word boundaries
        if len(sentence) > max_chars:
            if current_chunk:
                chunks.append(current_chunk.strip())
                current_chunk = ""
            # Sub-split long sentences at commas, semicolons, or word boundaries
            chunks.extend(_split_long_sentence(sentence, max_chars))
            continue
        
        if len(current_chunk) + len(sentence) + 1 > max_chars:
            chunks.append(current_chunk.strip())
            current_chunk = sentence
        else:
            current_chunk = f"{current_chunk} {sentence}" if current_chunk else sentence
    
    if current_chunk.strip():
        chunks.append(current_chunk.strip())
    
    return chunks
```

**Why 3800 not 4096?** Safety margin. Some TTS APIs count characters differently (e.g., multi-byte Unicode). 3800 gives us headroom without wasting capacity.

### Audio Generation and Concatenation

```
For each chunk:
  → Call tts.py with chunk text → /tmp/link2audio_{job_id}_chunk_{i}.mp3
  → Verify output file exists and is > 0 bytes

After all chunks:
  → Create ffmpeg concat file: /tmp/link2audio_{job_id}_filelist.txt
  → ffmpeg -f concat -safe 0 -i filelist.txt -c copy output.mp3
  → [Phase 3] If total size > platform limit: compress with ffmpeg (lower bitrate)
  → Clean up chunk files
```

**Why `-c copy` (stream copy)?** All chunks come from the same TTS engine with identical encoding parameters. Stream copy is instant and lossless — no re-encoding needed.

### Parallel TTS Generation (Phase 2)

For articles with many chunks (>5), generate TTS in parallel (up to 3-4 concurrent requests) to reduce wall-clock time. Use `asyncio.gather` with a semaphore.

```python
MAX_CONCURRENT_TTS = 4  # Respect rate limits

async def generate_all_chunks(chunks, voice, backend, job_id):
    semaphore = asyncio.Semaphore(MAX_CONCURRENT_TTS)
    
    async def generate_one(i, chunk):
        async with semaphore:
            return await generate_tts_chunk(chunk, voice, backend, job_id, i)
    
    tasks = [generate_one(i, c) for i, c in enumerate(chunks)]
    return await asyncio.gather(*tasks, return_exceptions=True)
```

### Rate Limit Handling

async def generate_tts_chunk(chunk, voice, backend, job_id, chunk_index,
                              max_retries=3, base_delay=2.0):
    for attempt in range(max_retries):
        try:
            result = await call_tts_api(chunk, voice, backend)
            return result
        except RateLimitError as e:
            if attempt == max_retries - 1:
                raise
            # Use Retry-After header if available, otherwise exponential backoff
            delay = e.retry_after if hasattr(e, 'retry_after') else base_delay * (2 ** attempt)
            logger.warning(f"Rate limited on chunk {chunk_index}, retrying in {delay}s")
            await asyncio.sleep(delay)
        except TtsApiError as e:
            if attempt == max_retries - 1:
                raise
            await asyncio.sleep(base_delay * (2 ** attempt))
```

### Rationale

- trafilatura handles boilerplate removal, ad stripping, and metadata extraction out of the box. For the vast majority of news articles, blog posts, and documentation pages, it produces clean text in under a second.
- Agent-browser (Playwright) is expensive and slow. Using it as the default — as one analyst initially proposed — would impose unnecessary latency and resource cost on the ~90% of requests that don't need it.
- The hybrid approach provides the speed of a lightweight library for the common case and the power of a full browser for the complex case.
- trafilatura does have known limitations (JS-rendered SPAs, some forum layouts, sites with heavy in-body ad injection). The fallback path directly addresses these.

### Extraction Pipeline

```
URL → Validate URL format
  → HTTP fetch (requests, User-Agent set, 15s timeout)
  → trafilatura.extract(html, include_comments=False, include_tables=False)
  → Evaluate result quality (see below)
  → If quality check fails:
      → [Phase 2] agent-browser fallback: Playwright fetch → rendered HTML → trafilatura.extract()
      → If still fails: return error to user
  → Clean text (normalize whitespace, strip artifacts)
  → Return (title, text, word_count, estimated_audio_duration)
```

### Fallback Trigger Heuristic

Rather than a single arbitrary character threshold, use a multi-factor check:

```python
def extraction_is_adequate(text: str, html: str) -> bool:
    if text is None:
        return False
    if len(text) < 200:
        return False
    # Check for paywall/login indicators in the original HTML
    paywall_signals = ["subscribe to continue", "sign in to read", "create a free account"]
    html_lower = html.lower()
    if any(signal in html_lower for signal in paywall_signals) and len(text) < 500:
        return False
    # Check content-to-boilerplate ratio (rough heuristic)
    if len(text) < len(html) * 0.02:  # Less than 2% of HTML is content
        return False
    return True
```

This is more robust than a single 200- or 250-character threshold while remaining simple and fast.

### Graceful Degradation When Agent-Browser Is Unavailable

- Check for agent-browser availability at startup; set a feature flag.
- If unavailable at request time, skip the fallback entirely.
- Inform the user: *"📄 I couldn't extract content from this page — it may require JavaScript to load. Try a different link?"*
- Log the URL for later analysis to identify patterns of sites that consistently need the fallback.

### Handling Paywalled Content (Best Effort)

- Set `User-Agent` to a Googlebot-like string (many paywalls serve full content to crawlers).
- If content is clearly truncated with paywall signals detected, inform the user: *"This article appears to be paywalled. I extracted what I could (~X words)."*
- Do **not** attempt aggressive paywall circumvention. Agent-browser alone does not bypass paywalls requiring authentication — this should not be overstated.

## 2. Text Chunking and Audio Concatenation

### The Problem

OpenAI TTS has a **4096 character limit** per request. A typical long-form article is 5,000–15,000 characters; some exceed 30,000.

### Chunking Strategy

**Chunk at sentence boundaries, targeting ~3800 characters per chunk** (leaving buffer for multi-byte Unicode edge cases).

```python
import re

def chunk_text(text: str, max_chars: int = 3800) -> list[str]:
    """Split text into chunks at sentence boundaries."""
    # First split into paragraphs, then sentences within paragraphs
    paragraphs = text.split('\n\n')
    sentences = []
    for para in paragraphs:
        para_sentences = re.split(r'(?<=[.!?])\s+', para.strip())
        sentences.extend(para_sentences)
        sentences.append("")  # Paragraph break marker

    chunks = []
    current_chunk = ""

    for sentence in sentences:
        if sentence == "":
            # Preserve paragraph breaks as minor pauses
            current_chunk += "\n\n"
            continue

        # If a single sentence exceeds max_chars, split at clause/word boundaries
        if len(sentence) > max_chars:
            if current_chunk.strip():
                chunks.append(current_chunk.strip())
                current_chunk = ""
            chunks.extend(_split_long_sentence(sentence, max_chars))
            continue

        if len(current_chunk) + len(sentence) + 1 > max_chars:
            chunks.append(current_chunk.strip())
            current_chunk = sentence
        else:
            current_chunk = f"{current_chunk} {sentence}" if current_chunk.strip() else sentence

    if current_chunk.strip():
        chunks.append(current_chunk.strip())

    return chunks


def _split_long_sentence(sentence: str, max_chars: int) -> list[str]:
    """Split an oversized sentence at clause boundaries, then word boundaries."""
    # Try clause boundaries first
    clauses = re.split(r'(?<=[,;:])\s+', sentence)
    sub_chunks = []
    current = ""
    for clause in clauses:
        if len(clause) > max_chars:
            if current.strip():
                sub_chunks.append(current.strip())
                current = ""
            # Last resort: split at word boundaries
            words = clause.split()
            for word in words:
                if len(current) + len(word) + 1 > max_chars:
                    sub_chunks.append(current.strip())
                    current = word
                else:
                    current = f"{current} {word}" if current else word
        elif len(current) + len(clause) + 1 > max_chars:
            sub_chunks.append(current.strip())
            current = clause
        else:
            current = f"{current} {clause}" if current else clause
    if current.strip():
        sub_chunks.append(current.strip())
    return sub_chunks
```

async def generate_tts_chunk_with_retry(chunk, voice, backend, job_id, chunk_index,
                                         max_retries=3, base_delay=2.0):
    for attempt in range(max_retries):
        try:
            return await call_tts_api(chunk, voice, backend, job_id, chunk_index)
        except RateLimitError as e:
            if attempt == max_retries - 1:
                raise
            delay = getattr(e, 'retry_after', base_delay * (2 ** attempt))
            logger.warning(f"Rate limited on chunk {chunk_index}, retrying in {delay}s")
            await asyncio.sleep(delay)
        except TtsApiError:
            if attempt == max_retries - 1:
                raise
            await asyncio.sleep(base_delay * (2 ** attempt))

### Audio Concatenation

```python
def concatenate_audio(chunk_paths: list[str], output_path: str) -> str:
    """Concatenate audio chunks using ffmpeg stream copy."""
    filelist_path = output_path.replace('.ogg', '_filelist.txt')
    with open(filelist_path, 'w') as f:
        for path in chunk_paths:
            f.write(f"file '{path}'\n")

    cmd = [
        'ffmpeg', '-y', '-f', 'concat', '-safe', '0',
        '-i', filelist_path, '-c', 'copy', output_path
    ]
    result = subprocess.run(cmd, capture_output=True, text=True)
    if result.returncode != 0:
        raise AudioConcatError(f"ffmpeg failed: {result.stderr}")

    # Cleanup
    os.remove(filelist_path)
    for path in chunk_paths:
        os.remove(path)

    return output_path
```

**Why `-c copy`?** All chunks come from the same TTS engine with identical encoding parameters (sample rate, bitrate, channels). Stream copy is instant and lossless. **Important prerequisite:** verify during testing that the TTS API returns consistent codec parameters across all chunks. If not, re-encode to a common format before concatenation.

## 3. Trigger Format

### Phase 1: Structured Commands

**Primary triggers:**
- `!listen <URL>` — canonical command
- `!read <URL>` — alias

**Phase 3 extensions:**
- `!listen <URL> voice:nova` — voice override
- `!listen <URL> backend:elevenlabs` — backend override

### Phase 2: Natural Language Detection

For casual users who paste a link with intent keywords:

```python
import re

EXPLICIT_TRIGGERS = [
    r'^[!/](listen|read|audio)\s+(https?://\S+)',
]

NL_TRIGGERS = [
    r'(?:read|listen to|convert|narrate|make audio)\s+(?:this\s+)?(?:article\s+)?(?:from\s+)?(https?://\S+)',
    r'(https?://\S+)\s+.*(?:read|listen|audio|voice note|narrate)',
]

def parse_trigger(message: str) -> dict | None:
    """Returns {url, voice, backend} or None."""
    # Check explicit commands first
    for pattern in EXPLICIT_TRIGGERS:
        match = re.search(pattern, message, re.IGNORECASE)
        if match:
            url = match.group(2)
            return {"url": url, "voice": extract_voice_override(message),
                    "backend": extract_backend_override(message)}

    # Natural language fallback (requires URL + intent keyword)
    for pattern in NL_TRIGGERS:
        match = re.search(pattern, message, re.IGNORECASE)
        if match:
            url = match.group(1)
            return {"url": url, "voice": None, "backend": None}

    return None
```

**Why both?** Explicit commands are unambiguous and fast — power users prefer them. Natural language fallback covers casual users. Requiring both a URL and an intent keyword for NL triggers avoids false positives.
