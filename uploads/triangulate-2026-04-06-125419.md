## 1. Architecture Overview

The `/debate` skill orchestrates a structured adversarial debate between three LLM providers. Unlike `/triangulate` (which generates independent analyses and merges them), `/debate` forces models to *defend their reasoning* against critiques from the other two models, producing outputs that have survived adversarial pressure-testing.

│  └─────────────────────────────────────────────────────────────┘    │

│  │ (Claude)  │  │ (GPT-4o)  │  │ (Gemini)  │                      │
│  └─────┬────┘  └─────┬────┘  └─────┬────┘                         │
│        │              │              │                               │
│  Stage 2: Cross-Critique Round 1                                    │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                         │

## 3. SKILL.md — Complete Content

```markdown
---

# /debate — Multi-Agent Structured Debate

## What This Skill Does

When the user invokes `/debate <task>`, you orchestrate a structured adversarial debate
between 3 different LLM providers (via LiteLLM proxy at localhost:4000). Each model
independently generates a response, then they critique each other's work across 2 rounds,
and finally a judge model synthesizes the consolidated result.

This differs from simple parallel generation (/triangulate) because models must **defend
their reasoning** against specific critiques, eliminating weak arguments and surfacing
genuine disagreements.

## Prerequisites

- LiteLLM proxy running at `http://localhost:4000` with access to:
  - Claude (claude-sonnet-4-20250514 or claude-opus-4-20250514)
  - GPT-4o (gpt-4o)
  - Gemini 2.5 Pro (gemini-2.5-pro)
- If any model is unavailable, the skill degrades gracefully (see Error Handling)

## Execution Protocol

You MUST follow these stages in exact order. Read `references/debate-protocol.md` for
the full protocol specification and `references/prompt-templates.md` for all prompts.

### Stage 0: Task Classification & Model Assignment

Classify the user's task into one of these categories and assign models accordingly.
Read `references/model-routing.md` for the full routing table.

Categories: `code`, `analysis`, `creative`, `factual`, `strategy`, `math`

### Stage 1: Independent Parallel Generation

Call all 3 models in parallel using `curl` to the LiteLLM proxy. Each model receives
the SAME prompt (the generation prompt from prompt-templates.md) with NO knowledge of
the other models' existence. This prevents anchoring bias.

**Critical**: Launch all 3 curl commands in background (`&`) and wait for all to complete.
Parse each response and save to variables/files.

### Stage 2: Cross-Critique Round 1

Each model receives the outputs of the OTHER TWO models and must produce structured
critiques. Again, run all 3 in parallel.

### Stage 3: Cross-Critique Round 2 (Rebuttal & Revision)

Each model receives the critiques that were directed at its original output, plus the
critiques it made of others. It must:
1. Rebut critiques it disagrees with (with reasoning)
2. Acknowledge valid critiques
3. Produce a REVISED response incorporating valid feedback

Run all 3 in parallel.

### Stage 4: Judge / Synthesis

A judge model (default: Claude, the strongest reasoning model) receives the FULL debate
transcript and produces the final consolidated result. Use the judge prompt from
prompt-templates.md.

### Stage 5: Output & Process Log

1. Present the final synthesized result to the user
2. Save the full debate log to `debate-logs/debate-YYYY-MM-DD-HHMMSS.md`

## How to Call LiteLLM

Use this exact curl pattern for ALL LLM calls:

```bash
curl -s http://localhost:4000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer sk-1234" \
  -d '{
    "model": "MODEL_NAME_HERE",
    "messages": [
      {"role": "system", "content": "SYSTEM_PROMPT_HERE"},
      {"role": "user", "content": "USER_PROMPT_HERE"}
    ],
    "temperature": TEMP_HERE,
    "max_tokens": MAX_TOKENS_HERE
  }' 2>/dev/null
```

To extract the response content from the JSON:
```bash
echo "$RESPONSE" | python3 -c "import sys,json; print(json.loads(sys.stdin.read())['choices'][0]['message']['content'])"
```

If python3 is unavailable, use:
```bash
echo "$RESPONSE" | grep -o '"content":"[^"]*"' | head -1 | sed 's/"content":"//;s/"$//'
```

**For parallel execution**, launch each curl in background and capture output:
```bash
# Launch all 3 in parallel
curl -s http://localhost:4000/v1/chat/completions ... > /tmp/debate_model_a.json &
PID_A=$!
curl -s http://localhost:4000/v1/chat/completions ... > /tmp/debate_model_b.json &
PID_B=$!
curl -s http://localhost:4000/v1/chat/completions ... > /tmp/debate_model_c.json &
PID_C=$!

# Wait for all to complete
wait $PID_A $PID_B $PID_C
```

## Error Handling

1. **Model unavailable**: If a model fails, retry once. If still failing, substitute
   with the next model in the fallback chain (see model-routing.md). If only 2 models
   are available, run a 2-model debate. If only 1, fall back to single-model with
   self-critique.
2. **Timeout**: Set curl timeout to 120 seconds (`--max-time 120`). LLM calls for
   debate rounds may be long.
3. **Malformed response**: If JSON parsing fails, retry the call once. If still
   malformed, log the raw response and skip that model for that round.
4. **LiteLLM proxy down**: Check connectivity first with a simple health check:
   `curl -s http://localhost:4000/health`. If down, inform the user and abort.

## Output Format

Present the final result in this structure:

```
## 🏛️ Debate Result

**Task**: [original task]
**Models**: [Model A], [Model B], [Model C]
**Judge**: [Judge model]
**Rounds**: 2

### Final Synthesized Answer

[The judge's consolidated result]

### Key Points of Agreement
- [point 1]
- [point 2]

### Resolved Disagreements
- [disagreement]: [resolution and reasoning]

### Unresolved Tensions
- [tension]: [why models couldn't agree, and judge's ruling]

---
*Full debate log saved to: debate-logs/debate-YYYY-MM-DD-HHMMSS.md*
*Total API calls: 11 | Estimated cost: [estimate]*
```

## Important Notes

- NEVER show raw JSON responses to the user — always parse and format
- ALWAYS run parallel stages actually in parallel (background processes), not sequentially
- The debate log file must contain the COMPLETE transcript of every stage
- If the task is simple/factual, consider telling the user that /debate may be overkill
  and a single model call would suffice — but still run it if they confirm
- Temperature settings vary by stage: generation=0.7, critique=0.3, judge=0.2
```

## 5. references/debate-protocol.md

```markdown
# Debate Protocol Specification

## Overview

The debate follows a structured 5-stage protocol. Each stage has specific inputs,
outputs, and constraints. This document defines the exact flow.

## Stage 0: Task Classification

**Input**: User's task description
**Process**: Claude (the orchestrator) classifies the task using heuristics from model-routing.md
**Output**: Task type string + model assignments for Voice A, Voice B, Voice C, Judge

No API call needed — the orchestrating Claude instance does this classification.

## Stage 1: Independent Parallel Generation

**Input**: User's task, wrapped in the generation prompt template
**Process**: 3 parallel API calls, one to each assigned model
**Output**: 3 independent responses (Response_A, Response_B, Response_C)

**Critical constraint**: Each model receives IDENTICAL prompts. No model knows the others
exist. This is what prevents anchoring bias — the key differentiator from approaches
where models see each other's work during generation.

**Parallel execution**: All 3 calls MUST be launched simultaneously using background
processes. Do NOT wait for one to finish before starting the next.

## Stage 2: Cross-Critique Round 1

**Input**: Each model receives the other two models' responses + the critique prompt
**Process**: 3 parallel API calls

Specifically:
- Model A receives Response_B and Response_C → produces Critique_A
- Model B receives Response_A and Response_C → produces Critique_B  
- Model C receives Response_A and Response_B → produces Critique_C

Each critique must address BOTH other responses, structured as:
1. Strengths of Response X
2. Weaknesses/errors in Response X
3. Strengths of Response Y
4. Weaknesses/errors in Response Y
5. Points where X and Y agree (and whether that agreement is correct)
6. Points where X and Y disagree (and which is more likely correct)

## Stage 3: Cross-Critique Round 2 (Rebuttal & Revision)

**Input**: Each model receives:
- Its own original response
- The critiques directed at its response (from the other two models)
- Its own critique of the others (for context)

**Process**: 3 parallel API calls

Each model must produce:
1. **Rebuttals**: For each critique of its work, either:
   - Acknowledge the critique as valid and explain how it will be addressed
   - Rebut the critique with specific reasoning for why the original was correct
2. **Revised Response**: An updated version of its original response incorporating
   valid feedback. Must explicitly mark what changed and why.

## Stage 4: Judge / Synthesis

**Input**: The COMPLETE debate transcript:
- Original task
- All 3 initial responses
- All 3 Round 1 critiques
- All 3 Round 2 rebuttals + revised responses

**Process**: Single API call to the judge model

**Output**: Structured synthesis containing:
1. Final consolidated answer
2. Key points of agreement (with confidence level)
3. Resolved disagreements (what was disputed, how it was resolved)
4. Unresolved tensions (genuine ambiguities or value-dependent choices)
5. Confidence assessment (how much the debate improved the answer)

## Stage 5: Output & Logging

**Process**:
1. Format the judge's synthesis for user presentation
2. Write the full debate log to `debate-logs/debate-YYYY-MM-DD-HHMMSS.md`
3. Present the formatted result to the user

## Timing Expectations

| Stage | Expected Duration | Notes |
|-------|-------------------|-------|
| Classification | <1 second | Local, no API call |
| Parallel Generation | 15-45 seconds | Bounded by slowest model |
| Critique Round 1 | 10-30 seconds | Shorter outputs |
| Critique Round 2 | 15-35 seconds | Medium outputs |
| Judge/Synthesis | 15-45 seconds | Long context input |
| **Total** | **~1-3 minutes** | Parallel execution is key |

## Degraded Mode: 2-Model Debate

If only 2 models are available:
- Run 2 parallel generations
- Each model critiques the other (not two others)
- Single round of rebuttal (since there's less diversity)
- Judge still synthesizes

## Degraded Mode: 1-Model Self-Debate

If only 1 model is available:
- Generate initial response
- Ask the model to critique its own response from 3 different perspectives:
  (devil's advocate, domain expert, end user)
- Ask the model to revise based on self-critique
- Skip judge stage (the revision IS the final output)
- Warn the user that this provides much less value than multi-model debate
```

# Prompt Templates for /debate

## Stage 1: Generation Prompt

### System Prompt
```
You are an expert analyst providing your independent assessment. Give your best, most
thorough response to the task below. Be specific, provide reasoning for your claims,
and flag any uncertainties or assumptions you're making.

Structure your response clearly with sections/headers as appropriate for the task type.
Do not hedge excessively — take clear positions where the evidence supports them, and
note where you're less certain.
```

### User Prompt
```
## Task

{{TASK_DESCRIPTION}}

## Instructions

Provide your complete, independent response to this task. Be thorough and specific.
Structure your response with clear sections. For any claims or recommendations, explain
your reasoning.
```

---

## Stage 2: Cross-Critique Round 1 Prompt

### System Prompt
```
You are a rigorous peer reviewer. You have been given two responses to the same task,
written by other analysts. Your job is to critically evaluate both responses.

Be specific in your critiques — point to exact claims, identify logical gaps, factual
errors, missing considerations, and unstated assumptions. Also acknowledge genuine
strengths. Do not be contrarian for its own sake; only raise critiques you believe
are substantive.
```

### User Prompt
```
## Original Task

{{TASK_DESCRIPTION}}

## Response from Analyst X

{{RESPONSE_X}}

## Response from Analyst Y

{{RESPONSE_Y}}

## Your Critique Assignment

Critically evaluate both responses above. For EACH response, address:

1. **Strengths**: What does this response get right? What is particularly insightful?
2. **Weaknesses**: What errors, gaps, or questionable claims do you identify? Be specific.
3. **Missing considerations**: What important aspects does this response overlook?
4. **Unstated assumptions**: What assumptions is this response making that should be examined?

Then address the responses together:

5. **Agreement analysis**: Where do both responses agree? Is that agreement well-founded?
6. **Disagreement analysis**: Where do they disagree? Which position is stronger and why?
7. **Synthesis opportunity**: What would an ideal response take from each?
```

---

## Stage 3: Rebuttal & Revision Prompt (Round 2)

### System Prompt
```
You are an analyst who has received peer review feedback on your work. You must engage
honestly with the critiques: acknowledge valid points and revise your work accordingly,
but also defend your positions where you believe the critiques are wrong.

Do not capitulate to critiques simply because they were raised. Defend your reasoning
where it is sound. But genuinely update your views where the critiques reveal errors
or gaps.
```

### User Prompt
```
## Original Task

{{TASK_DESCRIPTION}}

## Your Original Response

{{OWN_ORIGINAL_RESPONSE}}

## Critique from Reviewer 1 (regarding your response)

{{CRITIQUE_OF_YOU_FROM_REVIEWER_1}}

## Critique from Reviewer 2 (regarding your response)

{{CRITIQUE_OF_YOU_FROM_REVIEWER_2}}

## Your Assignment

### Part 1: Rebuttals

For each substantive critique raised about YOUR response, either:
- **ACKNOWLEDGE**: "Valid point. [Explain what was wrong and how you'll fix it]"
- **REBUT**: "I disagree. [Explain specifically why your original position is correct]"

Address critiques from both reviewers.

### Part 2: Revised Response

Provide your REVISED response to the original task, incorporating the valid feedback.
Clearly mark what changed from your original with [REVISED] tags and briefly note why.

If nothing needs changing, state that explicitly with your reasoning.
```

---

## Stage 4: Judge / Synthesis Prompt

### System Prompt
```
You are a senior judge synthesizing the results of a structured debate between three
expert analysts. You have access to the complete debate transcript: initial responses,
cross-critiques, rebuttals, and revised responses.

Your job is to produce the BEST POSSIBLE answer to the original task by:
1. Identifying where the debate reached genuine consensus
2. Adjudicating disagreements based on the strength of arguments presented
3. Incorporating insights that emerged through the critique process
4. Noting any genuine unresolvable tensions (where reasonable experts would disagree)

Be authoritative but honest about uncertainty. The value of this debate process is that
weak arguments have been challenged — your synthesis should reflect only the arguments
that survived scrutiny.
```

### User Prompt
```
## Original Task

{{TASK_DESCRIPTION}}

---

## DEBATE TRANSCRIPT

### Stage 1: Independent Responses

#### Voice A ({{MODEL_A_NAME}})
{{RESPONSE_A}}

#### Voice B ({{MODEL_B_NAME}})
{{RESPONSE_B}}

#### Voice C ({{MODEL_C_NAME}})
{{RESPONSE_C}}

---

### Stage 2: Cross-Critiques (Round 1)

#### Voice A's Critique of B and C
{{CRITIQUE_A}}

#### Voice B's Critique of A and C
{{CRITIQUE_B}}

#### Voice C's Critique of A and B
{{CRITIQUE_C}}

---

### Stage 3: Rebuttals & Revised Responses (Round 2)

#### Voice A's Rebuttal and Revision
{{REBUTTAL_A}}

#### Voice B's Rebuttal and Revision
{{REBUTTAL_B}}

#### Voice C's Rebuttal and Revision
{{REBUTTAL_C}}

---

## YOUR SYNTHESIS ASSIGNMENT

Produce a final, consolidated response to the original task. Structure your output as:

### Final Answer
[Your synthesized response — this should be the best possible answer, drawing on the
strongest arguments from the debate]

### Consensus Points
[List the key points where all three voices agreed, with your confidence level]

### Resolved Disagreements
[For each significant disagreement, state what was disputed, summarize both sides,
and explain your ruling and why]

### Unresolved Tensions
[Any genuine tensions where reasonable experts would disagree — explain the tradeoffs]

### Debate Value Assessment
[Brief note on how much the debate process improved the answer compared to what any

process?]
```

---

## Extracting Critiques Directed at a Specific Model

When preparing the Round 2 (Rebuttal) prompt for Model A, you need to extract the
portions of Critique_B and Critique_C that are specifically about Model A's response.

Since the critique prompt asks reviewers to address each response separately with clear
headers, you can extract the relevant sections. However, if extraction is difficult,
it is acceptable to provide the FULL critique from each reviewer and instruct the model
to focus on the parts about its own response. The rebuttal prompt handles this:

"For each substantive critique raised about YOUR response..."

This means the model will self-select the relevant critiques. This is simpler and
more robust than trying to parse critique sections programmatically.
```

This is a strong, comprehensive design document. The author has clearly thought through the end-to-end flow, from architecture to specific prompt templates. The inclusion of fallback modes, detailed logging, and packaging considerations is excellent.

### Overall Assessment

**Strengths:**
*   Well-structured and logical pipeline.
*   Excellent prompt engineering with clear instructions for each stage.
*   Robustness is considered with degraded modes and error handling.
*   Portability is a core design principle, evident in the use of `curl` and fallback parsing methods.

**Areas for Improvement:**
The design is very optimistic about the reliability and consistency of both the execution environment and the LLM outputs. The following points highlight areas where real-world friction could occur.

---

### 1. Practical Implementation Issues

*   **Race Condition in Parallel Execution:** The `SKILL.md` suggests writing parallel outputs to static filenames like `/tmp/debate_model_a.json`. If two users run `/debate` simultaneously, or one user runs it twice in quick succession, these processes will overwrite each other's temporary files, leading to corrupted data and failed runs.
    *   **Recommendation:** The design should specify using `mktemp` or including the process ID (`$$` in bash) in filenames to guarantee uniqueness for each parallel run.

*   **Complex Orchestration Logic:** The shell script logic required to manage 11 API calls, handle partial failures, parse multiple JSONs, and assemble massive context strings for later stages is non-trivial. The design assumes a powerful orchestrating agent (Claude) can flawlessly generate and execute this complex script. A single syntax error in the generated bash script could cause the entire process to fail opaquely.
    *   **Recommendation:** The design should acknowledge the complexity and perhaps suggest pre-validating the generated script or using a more robust intermediate language if possible within the Claude Code environment.

*   **Maintenance of Model-Specific Information:** The `model-routing.md` hardcodes specific model names (e.g., `claude-opus-4-20250514`) and a cost table is implied in the log format. These will become outdated quickly. The design lacks a strategy for keeping this information current.
    *   **Recommendation:** Note the maintenance burden. Suggest that the skill should be able to query the LiteLLM `/models` endpoint to dynamically discover available models, and that cost data needs a clear update path.

*   **Implicit Dependency on `python3` or `sed`:** The `SKILL.md` provides two methods for parsing JSON. While good for portability, it creates an implicit dependency. If neither is available in a minimal NanoClaw container, the skill will fail.
    *   **Recommendation:** The `install.sh` script or prerequisites should explicitly check for the existence of at least one of these tools.

### 2. Edge Cases and Error Handling

*   **Content-Free or Refusal Responses:** The error handling covers API failures (e.g., 500 errors, timeouts) but not *successful* API calls that return useless content. What happens if a model responds with "I cannot answer this question" or a boilerplate refusal? The current design would treat this as a valid response, pass it to other models for critique, and pollute the entire debate.
    *   **Recommendation:** The design needs a validation step after each generation to check for common refusal patterns. If a refusal is detected, it should be treated as a model failure and trigger the fallback logic.

*   **Partial Failure in Parallel Stages:** The design mentions degrading to a 2-model debate if a model is unavailable from the start. It's less clear about what happens if a model fails *mid-debate*. For example, if in Round 2, Model C's rebuttal call fails, do you proceed to the Judge with incomplete information? Or do you retry?
    *   **Recommendation:** The protocol should explicitly define how to handle partial failures within a stage. For example: "If a model fails during a critique/rebuttal round, proceed to the next stage but clearly mark the missing input in the Judge's context."

*   **Exceeding Context Window Limits:** The Judge prompt concatenates the *entire* debate history. For a complex task with verbose responses, this could easily exceed the context window of the judge model, leading to a hard failure at the final, most critical stage. The design notes "long context" but doesn't quantify the risk.
    *   **Recommendation:** The design should include a pre-flight check before the Judge stage. This involves tokenizing the prepared context and, if it exceeds a safety margin (e.g., 90% of the model's limit), either failing gracefully with a warning or attempting a summarization of earlier stages before sending to the Judge.

*   **Missing Prompt for 1-Model Degraded Mode:** The `debate-protocol.md` specifies a clever 1-model self-critique mode, but the `prompt-templates.md` file does not include the necessary prompts for this path (e.g., the prompt to critique its own work from three different perspectives). This is a clear content gap.

### 3. Prompt Engineering Quality

*   **Potential for Groupthink/Politeness:** The prompts are excellent but may encourage overly "polite" critiques. All models are framed as "expert analysts" and "rigorous peer reviewers." There's no incentive for a model to take a truly adversarial, "red team" stance, which is often where the most valuable insights in a debate come from.
    *   **Recommendation:** Consider dynamically assigning a "Devil's Advocate" or "Red Teamer" role to one of the models (e.g., Voice C, the "challenger"). Its system prompt for the critique stage would be explicitly instructed to find flaws, challenge core assumptions, and argue the opposite case, even if it seems plausible.

*   **Parsing Brittle-ness in Rebuttal Stage:** The design correctly notes that programmatically parsing critiques to isolate comments for a specific model is brittle. The proposed solution is to give the full text to the model and let it figure it out. While robust, this is inefficient and relies on the model to be a perfect parser. It might miss a critique or address one not meant for it.
    *   **Recommendation:** The critique prompt (Stage 2) could be strengthened to enforce a more machine-parsable output format, for example, by requiring critiques to be wrapped in XML tags like `<critique_for model="A">...</critique_for>`. This would make the orchestrator's job of assembling the rebuttal prompt much more reliable.

### 4. Cost Optimization Opportunities

*   **No "Economy" Mode:** The design defaults to using the most powerful and expensive models (Opus, GPT-4o, Gemini 2.5). For many tasks, this is overkill. The 5-9x cost multiplier is significant, and users may want a cheaper option.
    *   **Recommendation:** The design should include an optional "fast" or "economy" mode. This could be a flag (`/debate --fast <task>`) that instructs the model router to use cheaper, faster models (e.g., Sonnet, GPT-4-Turbo, Gemini Flash).

*   **Fixed Two Rounds of Debate:** The design fixes the debate at two rounds, citing diminishing returns. However, for some simple tasks, one round might be sufficient. For highly complex, novel problems, a third round might be valuable.
    *   **Recommendation:** Consider making the number of rounds configurable (`/debate --rounds=1 <task>`). A single round would significantly reduce cost and latency.

*   **Critique/Rebuttal Models:** The critique and rebuttal stages may not require the same level of generative power as the initial response.
    *   **Recommendation:** The design could specify using smaller, faster models for the critique rounds (Stages 2 & 3) while retaining the powerful models for initial generation (Stage 1) and final synthesis (Stage 4).

### 5. Portability Concerns

*   **`install.sh` Ambiguity:** The standalone repo file structure includes an `install.sh` script, but the design doesn't specify what it does. For portability, its actions and dependencies (e.g., `bash`, `cp`, `mkdir`) must be clearly defined and should not make assumptions about the user's environment beyond a standard POSIX system.
    *   **Recommendation:** The design document should include a brief specification for `install.sh`, e.g., "The script will create the `~/.claude/skills/debate` directory and copy the skill files into it, after confirming with the user."

*   **LiteLLM Configuration:** The design assumes LiteLLM is running and configured. For a standalone user, this is a significant setup hurdle. The included `setup-litellm.md` is good, but the process can be complex (API keys, environment variables, Docker).
    *   **Recommendation:** The `README.md` for the standalone repo should have a "Quick Start" section that uses a minimal Docker command to get LiteLLM running with placeholder API keys, making the initial setup as frictionless as possible.

# /debate — Multi-Agent Structured Debate

## What This Skill Does

When the user invokes `/debate <task>`, you orchestrate a structured adversarial debate
between 3 different LLM providers (via LiteLLM proxy at localhost:4000). Each model
independently generates a response, then they critique each other's work across 2 rounds,
and finally a judge model synthesizes the consolidated result.

This differs from simple parallel generation (/triangulate) because models must **defend
their reasoning** against specific critiques, eliminating weak arguments and surfacing
genuine disagreements.

## Prerequisites

- LiteLLM proxy running at `http://localhost:4000` with access to:
  - Claude (claude-sonnet-4-20250514 or claude-opus-4-20250514)
  - GPT-4o (gpt-4o)
  - Gemini 2.5 Pro (gemini-2.5-pro)
- If any model is unavailable, the skill degrades gracefully (see Error Handling)

## Execution Protocol

You MUST follow these stages in exact order. Read `references/debate-protocol.md` for
the full protocol specification and `references/prompt-templates.md` for all prompts.

### Stage 0: Task Classification & Model Assignment

Classify the user's task into one of these categories and assign models accordingly.
Read `references/model-routing.md` for the full routing table.

Categories: `code`, `analysis`, `creative`, `factual`, `strategy`, `math`

### Stage 1: Independent Parallel Generation

Call all 3 models in parallel using `curl` to the LiteLLM proxy. Each model receives
the SAME prompt (the generation prompt from prompt-templates.md) with NO knowledge of
the other models' existence. This prevents anchoring bias.

**Critical**: Launch all 3 curl commands in background (`&`) and wait for all to complete.
Parse each response and save to variables/files.

### Stage 2: Cross-Critique Round 1

Each model receives the outputs of the OTHER TWO models and must produce structured
critiques. Again, run all 3 in parallel.

### Stage 3: Cross-Critique Round 2 (Rebuttal & Revision)

Each model receives the critiques that were directed at its original output, plus the
critiques it made of others. It must:
1. Rebut critiques it disagrees with (with reasoning)
2. Acknowledge valid critiques
3. Produce a REVISED response incorporating valid feedback

Run all 3 in parallel.

### Stage 4: Judge / Synthesis

A judge model (default: Claude, the strongest reasoning model) receives the FULL debate
transcript and produces the final consolidated result. Use the judge prompt from
prompt-templates.md.

### Stage 5: Output & Process Log

1. Present the final synthesized result to the user
2. Save the full debate log to `debate-logs/debate-YYYY-MM-DD-HHMMSS.md`

## How to Call LiteLLM

Use this exact curl pattern for ALL LLM calls:

```bash
curl -s http://localhost:4000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer sk-1234" \
  -d '{
    "model": "MODEL_NAME_HERE",
    "messages": [
      {"role": "system", "content": "SYSTEM_PROMPT_HERE"},
      {"role": "user", "content": "USER_PROMPT_HERE"}
    ],
    "temperature": TEMP_HERE,
    "max_tokens": MAX_TOKENS_HERE
  }' 2>/dev/null
```

To extract the response content from the JSON:
```bash
echo "$RESPONSE" | python3 -c "import sys,json; print(json.loads(sys.stdin.read())['choices'][0]['message']['content'])"
```

If python3 is unavailable, use:
```bash
echo "$RESPONSE" | grep -o '"content":"[^"]*"' | head -1 | sed 's/"content":"//;s/"$//'
```

**For parallel execution**, launch each curl in background and capture output:
```bash
# Launch all 3 in parallel
curl -s http://localhost:4000/v1/chat/completions ... > /tmp/debate_model_a_$$.json &
PID_A=$!
curl -s http://localhost:4000/v1/chat/completions ... > /tmp/debate_model_b_$$.json &
PID_B=$!
curl -s http://localhost:4000/v1/chat/completions ... > /tmp/debate_model_c_$$.json &
PID_C=$!

# Wait for all to complete
wait $PID_A $PID_B $PID_C
```

## Error Handling

1. **Model unavailable**: If a model fails, retry once. If still failing, substitute
   with the next model in the fallback chain (see model-routing.md). If only 2 models
   are available, run a 2-model debate. If only 1, fall back to single-model with
   self-critique.
2. **Timeout**: Set curl timeout to 120 seconds (`--max-time 120`). LLM calls for
   debate rounds may be long.
3. **Malformed response**: If JSON parsing fails, retry the call once. If still
   malformed, log the raw response and skip that model for that round.
4. **LiteLLM proxy down**: Check connectivity first with a simple health check:
   `curl -s http://localhost:4000/health`. If down, inform the user and abort.

# Debate Protocol Specification

## Overview

The debate follows a structured 5-stage protocol. Each stage has specific inputs,
outputs, and constraints. This document defines the exact flow.

## Stage 0: Task Classification

**Input**: User's task description
**Process**: Claude (the orchestrator) classifies the task using heuristics from model-routing.md
**Output**: Task type string + model assignments for Voice A, Voice B, Voice C, Judge

No API call needed — the orchestrating Claude instance does this classification.

## Stage 1: Independent Parallel Generation

**Input**: User's task, wrapped in the generation prompt template
**Process**: 3 parallel API calls, one to each assigned model
**Output**: 3 independent responses (Response_A, Response_B, Response_C)

**Critical constraint**: Each model receives IDENTICAL prompts. No model knows the others
exist. This is what prevents anchoring bias — the key differentiator from approaches
where models see each other's work during generation.

**Parallel execution**: All 3 calls MUST be launched simultaneously using background
processes. Do NOT wait for one to finish before starting the next.

## Stage 2: Cross-Critique Round 1

**Input**: Each model receives the other two models' responses + the critique prompt
**Process**: 3 parallel API calls

Specifically:
- Model A receives Response_B and Response_C → produces Critique_A
- Model B receives Response_A and Response_C → produces Critique_B  
- Model C receives Response_A and Response_B → produces Critique_C

Each critique must address BOTH other responses, structured as:
1. Strengths of Response X
2. Weaknesses/errors in Response X
3. Strengths of Response Y
4. Weaknesses/errors in Response Y
5. Points where X and Y agree (and whether that agreement is correct)
6. Points where X and Y disagree (and which is more likely correct)

## Stage 3: Cross-Critique Round 2 (Rebuttal & Revision)

**Input**: Each model receives:
- Its own original response
- The critiques directed at its response (from the other two models)
- Its own critique of the others (for context)

**Process**: 3 parallel API calls

Each model must produce:
1. **Rebuttals**: For each critique of its work, either:
   - Acknowledge the critique as valid and explain how it will be addressed
   - Rebut the critique with specific reasoning for why the original was correct
2. **Revised Response**: An updated version of its original response incorporating
   valid feedback. Must explicitly mark what changed and why.

## Stage 4: Judge / Synthesis

**Input**: The COMPLETE debate transcript:
- Original task
- All 3 initial responses
- All 3 Round 1 critiques
- All 3 Round 2 rebuttals + revised responses

**Process**: Single API call to the judge model

**Output**: Structured synthesis containing:
1. Final consolidated answer
2. Key points of agreement (with confidence level)
3. Resolved disagreements (what was disputed, how it was resolved)
4. Unresolved tensions (genuine ambiguities or value-dependent choices)
5. Confidence assessment (how much the debate improved the answer)

## Stage 5: Output & Logging

**Process**:
1. Format the judge's synthesis for user presentation
2. Write the full debate log to `debate-logs/debate-YYYY-MM-DD-HHMMSS.md`
3. Present the formatted result to the user

## Timing Expectations

| Stage | Expected Duration | Notes |
|-------|-------------------|-------|
| Classification | <1 second | Local, no API call |
| Parallel Generation | 15-45 seconds | Bounded by slowest model |
| Critique Round 1 | 10-30 seconds | Shorter outputs |
| Critique Round 2 | 15-35 seconds | Medium outputs |
| Judge/Synthesis | 15-45 seconds | Long context input |
| **Total** | **~1-3 minutes** | Parallel execution is key |

## Degraded Mode: 2-Model Debate

If only 2 models are available:
- Run 2 parallel generations
- Each model critiques the other (not two others)
- Single round of rebuttal (since there's less diversity)
- Judge still synthesizes

## Degraded Mode: 1-Model Self-Debate

If only 1 model is available:
- Generate initial response
- Ask the model to critique its own response from 3 different perspectives:
  (devil's advocate, domain expert, end user)
- Ask the model to revise based on self-critique
- Skip judge stage (the revision IS the final output)
- Warn the user that this provides much less value than multi-model debate

Be specific in your critiques — point to exact claims, identify logical gaps, factual
errors, missing considerations, and unstated assumptions. Also acknowledge genuine
strengths. Do not be contrarian for its own sake; only raise critiques you believe
are substantive.
```

1. **Strengths**: What does this response get right? What is particularly insightful?

## Extracting Critiques Directed at a Specific Model

When preparing the Round 2 (Rebuttal) prompt for Model A, you need to extract the
portions of Critique_B and Critique_C that are specifically about Model A's response.

Since the critique prompt asks reviewers to address each response separately with clear
headers, you can extract the relevant sections. However, if extraction is difficult,
it is acceptable to provide the FULL critique from each reviewer and instruct the model
to focus on the parts about its own response. The rebuttal prompt handles this:

"For each substantive critique raised about YOUR response..."

This means the model will self-select the relevant critiques. This is simpler and
more robust than trying to parse critique sections programmatically.
```

## 8. Error Handling — Complete Specification

```markdown
# Error Handling Matrix

## Pre-flight Checks (before any debate stage)

1. **LiteLLM Health Check**:
   ```bash
   HEALTH=$(curl -s --max-time 5 http://localhost:4000/health)
   ```
   - If fails: Print error message and abort the process.

2. **Model Availability Check**:
   - Before starting, verify that all required models are available via LiteLLM.
   - If a model is unavailable, follow the fallback chain specified in `model-routing.md`.

## During Execution

1. **Race Condition Management**:
   - Use unique filenames for temporary files to avoid data corruption in concurrent runs.
   - Example: `/tmp/debate_model_a_$$.json` where `$$` is the process ID.

2. **Timeouts**:
   - Set a timeout for each `curl` call to prevent hanging indefinitely (`--max-time 120`).

3. **Content-Free or Refusal Responses**:
   - After each generation, check for refusal patterns (e.g., "I cannot answer this question").
   - If detected, treat as a failure and trigger fallback logic.

4. **Partial Failures**:
   - If a model fails during critique or rebuttal, proceed with available data.
   - Clearly mark missing inputs in the judge's context and log the error.

5. **Exceeding Context Window**:
   - Before the Judge stage, tokenize the prepared context.
   - If it exceeds 90% of the model's limit, attempt summarization of earlier stages.

6. **JSON Parsing Errors**:
   - Retry once if parsing fails. If still unsuccessful, log the raw response and skip the model.

## Post-Execution

1. **Log Completeness**:
   - Ensure the debate log captures all stages, including errors and retries.
   - Use timestamps and unique identifiers for traceability.

2. **User Feedback**:
   - Inform the user of any degraded performance or missing data in the final output.
   - Provide a summary of errors encountered and actions taken.

## Maintenance and Updates

1. **Model Information**:
   - Regularly update `model-routing.md` with current model IDs and capabilities.
   - Consider querying LiteLLM dynamically for available models.

2. **Cost Data**:
   - Keep cost estimates up-to-date based on API pricing changes.

3. **Environment Checks**:
   - `install.sh` should verify the presence of `python3` or `sed` for JSON parsing.
   - Provide clear instructions for setting up LiteLLM in `setup-litellm.md`.

4. **Prompt Updates**:
   - Periodically review and refine prompts to ensure clarity and effectiveness.
   - Address any changes in model behavior or capabilities.

## Additional Considerations

- **Economy Mode**: Consider implementing a flag for a cheaper, faster model configuration.
- **Configurable Rounds**: Allow users to specify the number of debate rounds for flexibility.
- **Devil's Advocate Role**: Introduce a role to encourage more adversarial critiques.
```

This design document provides a comprehensive and actionable plan for implementing the `/debate` skill, addressing both the initial requirements and the critique points. It ensures robustness, flexibility, and clarity in execution while maintaining a focus on delivering high-quality, pressure-tested results.
